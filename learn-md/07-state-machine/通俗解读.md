# 用人话聊聊状态机：Raft 里真正"干活"的那个人

> 这篇文档是 [07-state-machine/README.md](./README.md) 的通俗伴读版。README 里有 FSMCallerImpl 的 Disruptor 架构、批量合并策略、IteratorImpl 逐行分析，这篇文档帮你**先理解"为什么要这样设计"**。

---

## 开场：前面那些都是"准备工作"

如果你从第 1 篇读到这里，你可能已经有点累了——选举、日志复制、日志存储、快照……这么多组件，到底是为了什么？

让我做一个类比。Raft 集群就像一家**大型连锁饭店**：

- **选举** = 选出谁当主厨（Leader）
- **日志复制** = 主厨把点菜单抄给所有后厨（Follower）
- **日志存储** = 把点菜单写到账本上，防止丢失
- **快照** = 定期给厨房的"菜品库存"拍照片，这样新厨师不用从第一天的订单开始回放
- **BallotBox** = 确认大多数后厨都收到了点菜单

**但这些都不是"做菜"本身。**

真正"做菜"的——把宫保鸡丁炒出来的、把客人要的红烧肉端上桌的——是**状态机（StateMachine）**。

状态机是用户自己实现的。框架负责选举、复制、持久化——但业务逻辑是你的。如果你的业务是一个 KV 数据库，状态机就是 `map.put("foo", "bar")`；如果是一个配置中心，状态机就是 `config.update("key", "newValue")`。

---

## 第一幕：FSMCaller——"传菜窗口"

但状态机不能随便被调用。想想看：

- BallotBox 确认多数派后说"日志 101 提交了"——在 Replicator 线程里
- SnapshotExecutor 说"该做快照了"——在 Timer 线程里
- 某个 Follower 安装快照后说"该加载快照了"——在 RPC 回调线程里
- Node 说"我当选 Leader 了"——在选举回调线程里

这四个"通知"全部需要状态机处理，但它们来自四个不同的线程。如果直接调用状态机的方法——**状态机就要处理并发了**。

但写并发安全的业务代码是非常痛苦的事情。大多数开发者（包括我）都不擅长写无 bug 的并发代码。

JRaft 的解决方案是 `FSMCaller`——**传菜窗口**。

所有线程都不能直接进厨房，它们只能把"菜单"放到传菜窗口上。厨房里只有一个厨师（单消费者线程），按顺序从窗口取菜单、做菜。

这个"传菜窗口"就是 **Disruptor 无锁队列**。

```
Replicator线程  ─── onCommitted(101) ─────┐
Timer线程       ─── onSnapshotSave() ──────┤
RPC回调线程     ─── onSnapshotLoad() ──────┤──→ [Disruptor 队列] ──→ [单消费者线程(FSM线程)]
选举回调线程    ─── onLeaderStart(6) ──────┤                              ↓
错误通知线程    ─── onError(e) ────────────┘                        StateMachine.onApply()
                                                                    StateMachine.onSnapshotSave()
                                                                    StateMachine.onLeaderStart()
                                                                    ...
```

### 为什么用 Disruptor 而不是 BlockingQueue？

`BlockingQueue`（如 `LinkedBlockingQueue`）底层用锁实现。在高并发场景下，多个生产者线程竞争同一把锁，性能会下降。

Disruptor 是一个**无锁**的环形缓冲区（RingBuffer），生产者之间用 CAS 竞争序号，消费者轮询等待。在 JRaft 这种"多个生产者（各种线程通知）+ 单个消费者（FSM 线程）"的场景下，Disruptor 的性能远超 BlockingQueue。

## 第二幕：批量合并——"5 张菜单合成 1 批炒"

这是 FSMCaller 最精妙的优化，也是面试常考点。

假设 BallotBox 连续通知了 5 次：

```
onCommitted(101)
onCommitted(102)
onCommitted(103)
onCommitted(104)
onCommitted(105)
```

如果每次通知都立刻调用一次 `StateMachine.onApply()`，就要调 5 次。每次调用都有一定的开销（创建 Iterator、记录 metrics、调用用户代码……）。

FSMCaller 的做法是**只在批次结束时才真正执行**：

1. 收到 `COMMITTED(101)` → 记下 `maxCommittedIndex = 101`，不处理
2. 收到 `COMMITTED(102)` → `maxCommittedIndex = 102`，不处理
3. 收到 `COMMITTED(103)` → `maxCommittedIndex = 103`，不处理
4. 收到 `COMMITTED(104)` → `maxCommittedIndex = 104`，不处理
5. 收到 `COMMITTED(105)` → `maxCommittedIndex = 105`，**这是批次最后一个（`endOfBatch=true`）→ doCommitted(105)！**

`doCommitted(105)` 一次性从 `lastAppliedIndex+1` 到 `105`，把所有日志取出来交给状态机。

**5 次通知，1 次执行。** 如果是 1000 次通知呢？还是 1 次执行。批量合并把吞吐量提升了一个数量级。

### 非 COMMITTED 事件怎么处理？

如果批次中间插入了一个非 COMMITTED 事件（比如 `SNAPSHOT_SAVE`）：

```
COMMITTED(101)
COMMITTED(102)
SNAPSHOT_SAVE ← 非 COMMITTED 事件！
COMMITTED(103)
COMMITTED(104)
```

FSMCaller 的规则是：**遇到非 COMMITTED 事件时，先把积压的 COMMITTED 处理完**。

所以处理顺序是：
1. `doCommitted(102)`（先处理积压的 101-102）
2. `doSnapshotSave()`（再处理快照保存）
3. `doCommitted(104)`（最后处理 103-104）

这保证了事件之间的**因果顺序**：快照保存一定发生在它之前的所有日志都被应用之后。

## 第三幕：Iterator——"菜单列表"

`doCommitted(105)` 确定了要处理的日志范围：从 `lastAppliedIndex+1` 到 `105`。但怎么把这些日志交给用户呢？

JRaft 设计了一个 `Iterator`（迭代器），让用户像遍历列表一样处理日志：

```java
public void onApply(Iterator iter) {
    while (iter.hasNext()) {
        ByteBuffer data = iter.getData();    // 获取日志数据
        applyToMyStateMachine(data);          // 执行业务逻辑
        if (iter.done() != null) {
            iter.done().run(Status.OK());     // 回调客户端"成功了"
        }
        iter.next();                          // 移动到下一条
    }
}
```

### `done()` 为什么可能为 null？

在 Leader 上，每条写日志都有一个对应的 `Closure`——客户端在等着回复。`done()` 返回的就是这个 `Closure`，调用 `done.run(OK)` 就是告诉客户端"你的写请求成功了"。

但在 Follower 上，日志是从 Leader 复制过来的，**没有对应的客户端等着**。所以 `done()` 返回 `null`。

还有一种情况：Leader 重启后重放旧日志——这些日志的客户端连接早就断了，也没有 `Closure`。

### `setErrorAndRollback()` 是什么？

如果用户在 `onApply` 中发现某条日志无法处理（比如数据格式错误），可以调用 `iter.setErrorAndRollback(n, status)`：

- `n`：回滚几条日志
- `status`：错误描述

调用后，这些日志不会被标记为"已应用"，对应的 `Closure` 会收到错误通知（客户端知道写失败了）。而且节点会进入 `STATE_ERROR` 状态——**这是一个严重事件**，意味着状态机无法继续工作。

### ⚠️ 必须消费完所有日志！

如果你在 `onApply` 中提前 `return`，没有消费完 Iterator 的所有条目——**那些没消费的日志的客户端 `Closure` 不会被调用，客户端会永远等待直到超时。**

JRaft 在 `doApplyTasks` 末尾会强制调用 `iter.next()`（避免真正的死循环），但这意味着那些被跳过的日志虽然被"标记为已应用"，实际上**业务逻辑没有执行**——状态机可能出现数据不一致。

## 第四幕：ClosureQueue——"回调排队"

每条日志可能带一个 `Closure`（客户端回调）。这些 `Closure` 怎么和日志对应起来？

JRaft 用 `ClosureQueue`——一个按 logIndex 顺序排列的队列：

```
firstIndex = 101
queue: [closure_101, closure_102, null, closure_104, closure_105]
                                  ↑ 这条日志是配置变更，没有客户端回调
```

当 `doCommitted(105)` 时，`popClosureUntil(105)` 一次性弹出 101-105 的所有 `Closure`，传给 `IteratorImpl`。Iterator 遍历日志时，通过 `currentIndex - firstClosureIndex` 索引到对应的 `Closure`。

### 为什么 `clear()` 在锁外调用 `done.run()`？

Leader 下台时（`stepDown`），需要通知所有等待中的客户端"Leader 变了，请重试"——调用所有 `Closure` 的 `done.run(EPERM)`。

但 `done.run()` 可能触发网络 IO（发送 RPC 响应），如果在锁内执行，会长时间持锁，阻塞其他操作。

JRaft 的做法是：在锁内用新的空列表替换旧列表（`this.queue = new LinkedList<>()`），然后**在锁外**遍历旧列表调用 `done.run()`。

这个模式在 JRaft 里随处可见——**"先在锁内收集，在锁外执行回调"**。

## 第五幕：错误处理——"厨房着火了"

状态机出错是非常严重的事件。一旦 `setError()` 被调用：

1. `this.error` 被设置为非空
2. `fsm.onError(e)` 通知用户："你的状态机出问题了"
3. `node.onError(e)` 通知节点：节点进入 `STATE_ERROR` 状态
4. 后续所有 `doCommitted()` 直接 return（不再应用任何日志）
5. 节点不再接受写请求、不参与选举

**这相当于"厨房着火了，关门停业"。** 必须人工介入修复状态机后，重启节点才能恢复。

### 错误来源有哪些？

1. `IteratorImpl.next()` 获取日志失败（日志损坏或被截断）→ `ERROR_TYPE_LOG`
2. 用户在 `onApply` 中调用 `setErrorAndRollback()` → `ERROR_TYPE_STATE_MACHINE`
3. `onSnapshotLoad` 返回 `false`（快照加载失败）→ `ERROR_TYPE_STATE_MACHINE`
4. 快照元数据读取失败且是磁盘 IO 错误 → `ERROR_TYPE_SNAPSHOT`
5. `SnapshotExecutorImpl.reportError()` → `ERROR_TYPE_SNAPSHOT`

### setError() 的防重复设计

`setError()` 内部检查 `error.getType() != ERROR_TYPE_NONE`——如果已经有错误了，就不会再次调用 `fsm.onError()` 和 `node.onError()`。保证用户的 `onError()` 回调只被调用一次。

## 第六幕：onApply 中的一个隐藏陷阱

`doApplyTasks()` 方法里，调用 `fsm.onApply(iter)` 时**没有 try-catch**！

这意味着如果你的 `onApply` 抛出了一个未捕获的 `RuntimeException`（比如 `NullPointerException`）：

1. 异常被 Disruptor 的 `LogExceptionHandler` 捕获并打印日志
2. **但 `setError()` 不会被调用！** 节点继续运行
3. 这批日志的 `Closure` 不会被调用——客户端超时
4. 状态机可能处于不一致状态（部分日志被应用了，部分没有）

**这是 JRaft 的一个设计选择**：不替你决定"RuntimeException 应该导致节点停机"。如果你希望 RuntimeException 也触发节点停机，需要在 `onApply` 中自己 catch 所有异常，然后调用 `iter.setErrorAndRollback()`。

最佳实践：

```java
public void onApply(Iterator iter) {
    while (iter.hasNext()) {
        try {
            ByteBuffer data = iter.getData();
            applyToMyStateMachine(data);
            if (iter.done() != null) {
                iter.done().run(Status.OK());
            }
        } catch (Exception e) {
            // 出错了：回滚当前日志，通知框架
            iter.setErrorAndRollback(1, new Status(-1, e.getMessage()));
            return;
        }
        iter.next();
    }
}
```

## 第七幕：三个 Index 你分得清吗？

FSMCaller 维护了三个关键 index：

| Index | 含义 | 谁更新的 |
|-------|------|---------|
| `lastCommittedIndex` | BallotBox 通知的"已达多数派"的最大 index | BallotBox.commitAt() → onCommitted() |
| `applyingIndex` | 当前正在 apply 的 log index | IteratorImpl.next() |
| `lastAppliedIndex` | 状态机已成功 apply 的最大 index | doCommitted() 末尾 |

正常情况下：`lastAppliedIndex ≤ applyingIndex ≤ lastCommittedIndex`

它们的差值代表"积压量"：
- `lastCommittedIndex - lastAppliedIndex` 大 → 状态机处理慢，日志积压
- `applyingIndex` 在快速追赶 `lastCommittedIndex` → 状态机正在批量处理

如果 `lastCommittedIndex` 远大于 `lastAppliedIndex` 且长时间不收敛——你的 `onApply` 太慢了，需要优化。

## 总结

状态机和 FSMCaller 的核心设计：

| 设计 | 解决的问题 |
|------|-----------|
| Disruptor 无锁队列 | 多生产者高性能异步通知 |
| 单消费者线程 | 状态机不需要并发安全 |
| COMMITTED 批量合并 | 减少 onApply 调用次数，提升吞吐 |
| ClosureQueue 按序管理 | Closure 和日志的对应关系不出错 |
| IteratorImpl | 用户友好的批量日志遍历 API |
| setError 防重复 | onError 只调用一次 |
| 锁外执行回调 | 防止持锁执行 IO 导致死锁 |

**一句话总结**：FSMCaller 是 Raft "日志共识"和"业务执行"之间的桥梁——Disruptor 队列 + 单线程 + 批量合并，让用户写出简单的单线程 `onApply`，就能享受到分布式一致性的保证。
