# 用人话聊聊成员变更：不停机换人的学问

> 这篇文档是 [09-membership-change/README.md](./README.md) 的通俗伴读版。README 里有联合共识的完整状态机、ConfigurationCtx 源码分析和 Learner 机制，这篇文档帮你**理解为什么"换人"这么难**。

---

## 开场：一个看似简单的需求

"集群跑着 3 台机器，我想加一台，变成 4 台。"

听起来多简单？改个配置文件，告诉集群"现在有 4 台了"——搞定？

**大错特错。** 成员变更是分布式系统里**最容易出事**的操作之一。无数生产事故都是在"加减节点"的时候发生的。

为什么？让我讲个故事。

---

## 第一幕：脑裂的噩梦

假设集群有 5 台机器：A、B、C、D、E。多数派 = 3。

现在你要把配置从 {A,B,C,D,E}（5 台，多数派=3）改成 {A,B,C,F,G}（5 台，多数派=3）——把 D、E 换成 F、G。

如果你"一步到位"直接切换配置——

- A、B、C 先收到新配置，开始用 {A,B,C,F,G}
- D、E 还没收到新配置，还在用 {A,B,C,D,E}

这时候：
- {A,B,C} 是新配置的多数派 → 它们可以选出一个 Leader
- {A,D,E} 是旧配置的多数派 → 它们也可以选出一个 Leader

**两个 Leader 同时存在——脑裂！**

你看，问题的根源是：**新旧两个配置可以各自独立地形成多数派。** 这就像两个政府同时发号施令，国家就乱了。

---

## 第二幕：联合共识——"过渡委员会"

Raft 论文提出了一个优雅的解决方案：**Joint Consensus（联合共识）**。

思路是不搞"一步切换"，而是搞一个**过渡期**。在过渡期里，**任何决策都需要新旧两个配置都同意**。

打个比方：公司要换董事会。不能一下子全换——先成立一个"过渡委员会"，任何重大决策既要旧董事会过半同意、也要新董事会过半同意。等过渡期结束，旧董事会正式解散，新董事会接管。

在过渡期内：
- 旧配置的多数派不能独立做决策（因为还需要新配置也同意）
- 新配置的多数派也不能独立做决策（因为还需要旧配置也同意）
- **只有同时满足新旧两个多数派的提案才能通过**

这样就不可能出现"两个 Leader"了——任何 Leader 的产生都需要新旧两方的共同认可。

### 两阶段流程

1. **Phase 1：写入联合配置 `C_old,new`**
   - Leader 写一条配置日志：`{peers: [A,B,C,F,G], oldPeers: [A,B,C,D,E]}`
   - 从这一刻起，所有操作（选举、日志提交）都需要**两组 quorum**
   - `C_old,new` 被多数派确认后（需要旧多数派和新多数派都确认），进入 Phase 2

2. **Phase 2：写入新配置 `C_new`**
   - Leader 写一条配置日志：`{peers: [A,B,C,F,G]}`（没有 `oldPeers`）
   - `C_new` 被新配置的多数派确认后，旧配置正式作废
   - D 和 E 不再是集群成员，它们会自动退出（收到不包含自己的配置后 stepDown）

### Ballot 的双计票器

还记得我们在日志复制那篇讲过的 `Ballot` 吗？它内部有两个计票器：`quorum` 和 `oldQuorum`。

- 正常情况下只用 `quorum`：peers 的多数派
- Joint Consensus 期间同时用 `quorum` 和 `oldQuorum`：peers 的多数派 AND oldPeers 的多数派

`isGranted()` 的判断是：`quorum <= 0 && oldQuorum <= 0`——**两组都过半才算通过**。

## 第三幕：JRaft 的简化——addPeer/removePeer

Raft 论文的 Joint Consensus 支持"一次性换多个节点"。但 JRaft 提供了一组更常用的 API：

- `addPeer(peer)`：加一个节点
- `removePeer(peer)`：删一个节点
- `changePeers(newPeers)`：换一批节点（走 Joint Consensus）

`addPeer` 和 `removePeer` 本质上也是走 Joint Consensus（两阶段配置日志），只是被封装成了更简单的接口。

### addPeer 的完整流程

假设集群是 {A,B,C}，要加 D。

1. **检查**：D 是否已经在配置里了？是 → 直接返回
2. **先把 D 当 Learner 追日志**：D 加入集群后，日志可能落后很多。如果直接算入 quorum，D 还没追上日志就参与投票——可能导致可用性下降。先让 D 以 Learner 身份追日志（Learner 不算入 quorum，不影响集群可用性）
3. **等 D 追上**：Replicator 把日志发给 D，等 D 的日志追到和 Leader 差不多
4. **写 Joint 配置**：`{peers: [A,B,C,D], oldPeers: [A,B,C]}`
5. **写 New 配置**：`{peers: [A,B,C,D]}`
6. **完成**：D 正式成为集群成员

## 第四幕：Learner——"试用期员工"

Learner 是 JRaft 对 Raft 的一个重要扩展。

Learner 只接收日志复制，**不参与投票和 quorum 计数**。它就像公司里的"试用期员工"——可以旁听会议、接收文件，但没有投票权。

Learner 的用途：

1. **新节点预热**：新节点加入前先当 Learner 追日志，追上后再正式加入
2. **只读副本**：某些节点只负责读请求，不参与写。设为 Learner 可以减少 quorum 大小（3 节点 + 2 Learner 的 quorum 还是 2，而不是 3）
3. **跨机房备份**：远程机房的节点设为 Learner，异步接收数据，不影响本地机房的写延迟

### Learner 和 Follower 的区别

| | Follower | Learner |
|---|---|---|
| 接收日志复制 | ✅ | ✅ |
| 参与投票 | ✅ | ❌ |
| 计入 quorum | ✅ | ❌ |
| 能被选为 Leader | ✅ | ❌ |
| 配置变更需要同意 | ✅ | ❌ |

## 第五幕：ConfigurationCtx——"状态机中的状态机"

成员变更的两阶段流程，在 JRaft 里由 `ConfigurationCtx` 管理。它本身就是一个小型状态机：

```
STAGE_NONE → STAGE_CATCHING_UP → STAGE_JOINT → STAGE_STABLE → STAGE_NONE
```

- **STAGE_NONE**：空闲，没有进行中的配置变更
- **STAGE_CATCHING_UP**：新节点正在追日志（Learner 模式）
- **STAGE_JOINT**：Joint 配置已写入，等待确认
- **STAGE_STABLE**：新配置已写入，等待确认

**关键不变式：同一时刻只能有一个配置变更在进行。** 如果你在 STAGE_JOINT 时又发起一个 addPeer——被拒绝（EBUSY）。

为什么？因为并发的配置变更会让系统陷入不可预测的状态——两组 Joint 配置叠加在一起，谁也说不清"当前的 quorum 到底是多少"。

### no-op 日志的隐藏作用

还记得 Leader 当选后要提交一条空日志（no-op）吗？这条 no-op 日志还有一个重要作用：**重新激活未完成的配置变更**。

如果前任 Leader 正在做配置变更（比如写了 Joint 配置还没来得及写 New 配置就挂了），新 Leader 当选后需要"继续完成"这个配置变更。`becomeLeader()` 中调用 `confCtx.flush()` 就是干这个的——检查当前配置是否处于 Joint 状态，如果是，继续写 New 配置。

## 第六幕：那些坑

### 坑 1：加节点时忘记等追日志

如果新节点还没追上日志就直接加入 quorum：
- 3 节点集群变 4 节点，quorum 从 2 变成 3
- 新节点还没追上 → 只有原来的 3 台能投票
- 如果原来的 3 台有 1 台挂了 → 只剩 2 台能投票 → 达不到 quorum 3 → **集群不可用！**

JRaft 的 `addPeer` 默认会先让新节点追日志，追上后再正式加入。但如果你用 `changePeers` 一次性换多个节点，需要自己确保新节点已经准备好了。

### 坑 2：removePeer 自己

如果 Leader 把自己从配置里移除了——新配置里没有自己——Leader 会在 New 配置提交后主动 `stepDown()`。

这个过程是安全的，但会导致一次额外的选举。如果你在做"替换 Leader"的操作，建议先 `transferLeadership` 到另一台，再 `removePeer` 旧 Leader。

### 坑 3：配置变更期间 Leader 挂了

如果 Joint 配置写了但 New 配置还没写，Leader 就挂了。新 Leader 当选后，会检查配置状态：

- 如果 Joint 配置已提交 → 继续写 New 配置
- 如果 Joint 配置没提交 → 会被新 Leader 的日志覆盖，相当于配置变更被回滚

Raft 的安全性保证了这两种情况都不会导致不一致。

---

## 总结

成员变更的核心设计：

| 设计 | 解决的问题 |
|------|-----------|
| Joint Consensus | 防止新旧配置各自形成多数派导致脑裂 |
| 两阶段配置日志 | 安全地从旧配置过渡到新配置 |
| Ballot 双计票器 | Joint 期间同时满足新旧 quorum |
| Learner | 新节点预热 + 只读副本 + 跨机房备份 |
| ConfigurationCtx 状态机 | 保证同一时刻只有一个配置变更 |
| addPeer 先追日志 | 防止新节点拖累集群可用性 |
| no-op 日志续接配置变更 | 新 Leader 接手未完成的配置变更 |

**一句话总结**：成员变更的核心挑战是"不能让新旧配置各自为政"。Joint Consensus 用"过渡委员会"的机制，保证了任何决策都需要新旧双方共同认可，从而安全地完成换人。
