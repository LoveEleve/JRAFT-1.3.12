# 用人话聊聊 Placement Driver：图书馆的"馆长"

> 这篇文档是 [S7-Placement-Driver.md](./S7-Placement-Driver.md) 的通俗伴读版。

---

## 开场：为什么需要一个"馆长"？

上一篇我们聊了 RheaKV——把数据分成很多 Region，每个 Region 用一个 Raft Group 管理。但有几个问题还没回答：

- **谁来决定 Region 怎么分？** Region 数据太多了该怎么拆？
- **谁来决定 Region 放在哪？** 某台机器负载太高了，该把 Region 迁到哪里？
- **谁来平衡 Leader 分布？** 如果所有 Region 的 Leader 都在同一台机器上，这台机器的压力就太大了

**Placement Driver（PD）就是那个"图书馆馆长"**——负责全局的调度和管理。它不直接参与数据的读写，而是做"上帝视角"的决策。

---

## 第一幕：PD 的三大职责

### 1. 维护集群拓扑

PD 知道整个集群的"地图"：
- 有哪些 Store（物理节点）
- 每个 Store 上有哪些 Region
- 每个 Region 的 Leader 在哪里
- 每个 Region 的 key 范围是什么

这些信息通过 Store 定期上报的**心跳**来维护。每个 StoreEngine 每隔几秒就给 PD 发一次心跳："我是 Store-1，我有 Region-1、Region-2，Region-1 的数据量是 256MB……"

### 2. Region 自动分裂

当某个 Region 的数据量超过阈值（默认 512MB），PD 决定分裂它。

PD 不会自己动手——它把"分裂指令"通过心跳响应下发给对应 Store 的 RegionEngine。RegionEngine 收到指令后，通过 Raft 日志执行分裂。

为什么不让 PD 直接分裂？因为分裂操作必须走 Raft 共识——**所有副本必须在同一个日志位置执行分裂**，否则不同副本的 Region 范围就对不上了。

### 3. 负载均衡

PD 监控所有 Store 的负载：
- CPU 使用率
- 磁盘使用率
- Region 数量
- Leader 数量

如果某个 Store 的负载明显高于平均水平，PD 会触发**Region 迁移**（把某个 Region 的副本从高负载 Store 迁到低负载 Store）或 **Leader 转移**（把某个 Region 的 Leader 从高负载 Store 转到低负载 Store）。

---

## 第二幕：PD 本身怎么保证高可用？

PD 是集群的"大脑"——如果 PD 挂了怎么办？

答案是：**PD 本身也是一个 Raft Group！**

没错，PD 集群由 3 或 5 个节点组成，通过 JRaft 做共识。PD 的元数据（集群拓扑、Region 分布等）都通过 Raft 日志复制和持久化。

所以你可以说：**RheaKV 用 Raft 来管数据，PD 用 Raft 来管元数据——Raft 一路递归到底。**

---

## 第三幕：Fake PD vs Embedded PD

JRaft 的 PD 有两种部署模式：

### 1. Fake PD（内嵌式，默认）

PD 的功能直接嵌入到 RheaKV 的 Store 进程中。每个 Store 都有一份路由表的完整副本，通过互相通信来保持同步。

优点：不需要额外部署 PD 集群，简单方便。
缺点：分裂和均衡逻辑比较简单，不适合超大规模集群。

### 2. 独立部署的 PD

PD 作为独立的服务集群部署。Store 通过 RPC 与 PD 通信，上报心跳、接收调度指令。

优点：调度逻辑可以更复杂、更智能。
缺点：需要额外部署和维护 PD 集群。

---

## 第四幕：调度决策的执行流程

```
Store 心跳：我有 Region-1(300MB), Region-2(150MB)
    │
    ▼
PD 检查：Region-1 太大了(>256MB)，应该分裂
    │
    ▼
PD 通过心跳响应下发指令：
    "请在 key='m' 处分裂 Region-1"
    │
    ▼
Store 收到指令 → RegionEngine 执行：
    Region-1 [a, z) → Region-1 [a, m) + Region-3 [m, z)
    （通过 Raft 日志保证所有副本一致）
    │
    ▼
Store 下次心跳上报新的 Region 列表
    │
    ▼
PD 更新集群拓扑
```

---

## 总结

Placement Driver 的核心设计：

| 设计 | 解决的问题 |
|------|-----------|
| 心跳上报 | PD 掌握全局拓扑 |
| 分裂指令下发 | Region 数据量控制 |
| Leader 均衡 | 避免热点 Store |
| 负载均衡 | 避免资源倾斜 |
| PD 自身 Raft 高可用 | PD 不是单点故障 |
| Fake PD 模式 | 小集群零运维 |

**一句话总结**：PD 是 RheaKV 的"上帝视角"——不直接参与数据读写，但决定了数据怎么分、放在哪、Leader 在谁手里。
