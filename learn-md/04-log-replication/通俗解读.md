## 开场：选举结束了，真正的工作才刚开始

上一篇我们聊了选举——Leader 千辛万苦当选了。但当选只是开始，**Leader 真正的日常工作是日志复制**。

如果把 Raft 集群比作一个班级，选举只是"选班长"——一年可能就选一次。但日志复制是"班长每天把课堂笔记抄给所有同学"——这才是每天都在做的事情。而且这个"抄笔记"的过程远比你想象中复杂。

为什么复杂？因为在分布式系统里，**一切你能想到的坏事，迟早都会发生**：

- 网络可能延迟、丢包、乱序
- Follower 可能宕机、重启、日志落后
- Leader 可能刚发完消息就挂了
- 新旧 Leader 的日志可能有冲突

日志复制机制就是要在这些"坏事"不断发生的情况下，**保证所有节点的日志最终一致**。

好，让我们开始吧。

## 第一幕：一个写请求的完整旅程（写路径回顾）

在深入日志复制之前，让我们先回顾一下一个写请求的完整旅程：

```
客户端："把 key=foo 设为 bar"
    │
    ▼
① NodeImpl.apply()
    │ 检查：我是 Leader 吗？
    │ 把请求包装成 LogEntry
    ▼
② LogManager.appendEntries()
    │ 日志写入内存缓冲区
    │ 异步写入 RocksDB（磁盘）
    │ 通知 Replicator："有新日志了！"
    ▼
③ Replicator.sendEntries()                    ←── 本篇的重点！
    │ 通过 RPC 把日志发给 Follower
    ▼
④ Follower 收到日志
    │ 校验 + 写入本地
    │ 返回确认
    ▼
⑤ BallotBox.commitAt()                        ←── 本篇的重点！
    │ 统计：超过半数确认了吗？
    │ 超过了 → "提交！"
    ▼
⑥ FSMCaller.onCommitted()                     ←── 本篇的重点！
    │ 异步通知状态机
    ▼
⑦ StateMachine.onApply()
    │ 执行业务逻辑：map.put("foo", "bar")
    │ 回调客户端："成功了！"
```

这条路径里的 ①②⑦ 我们在前面几篇已经聊过了。本篇的重点是 **③④⑤⑥**——也就是"日志怎么从 Leader 到 Follower"和"怎么确认多数派"。

## 第二幕：Replicator——每个 Follower 的"专属抄写员"

### 为什么需要 Replicator？

Leader 需要把日志发给集群里的所有 Follower。但 Leader 不能自己一个个去发——那样代码太乱了，而且不同 Follower 的进度可能不同（有的追上了，有的落后很多）。

JRaft 的设计是：**每个 Follower 配一个 Replicator。** Replicator 就是 Leader 派出去的"专属抄写员"——"你只负责 Follower B，我只负责 Follower C"。每个抄写员独立工作、互不干扰。

### Replicator 的状态机

Replicator 有 5 个状态：

```
Created → Probe → Replicate ⇄ Snapshot
                      ↕
                  Destroyed
```

让我解释每个状态：

- **Created**：刚创建，还没开始工作
- **Probe（探测）**：不知道 Follower 的日志到了哪里，先发一个空请求"试探"一下
- **Replicate（复制）**：知道 Follower 的进度了，开始正式复制日志——这是性能最高的模式
- **Snapshot（快照安装）**：Follower 落后太多，需要发整个快照给它
- **Destroyed**：已销毁

**为什么要有 Probe 模式？**

新当选的 Leader 并不知道每个 Follower 的日志到了哪里。也许 Follower B 已经追到了第 100 条，也许 Follower C 只有第 50 条。Leader 得先"试探"一下：发一个空的 AppendEntries 请求，不带任何日志条目，只带"我这边的进度信息"。根据 Follower 的回复，Leader 就知道该从哪里开始发了。

就像你去帮朋友抄笔记——你不能上来就从第 1 页开始抄，得先问一句："你记到第几页了？"

## 第三幕：Pipeline——"不等签收就连续寄快递"

这是 JRaft 日志复制中**最精妙的性能优化**。

### 没有 Pipeline 的世界

假设你要给 Follower 发 100 条日志。最朴素的做法是：

1. 发第 1 条 → 等 Follower 确认 → 发第 2 条 → 等确认 → 发第 3 条 → ……

每条日志都要等一个网络往返（RTT）。如果 RTT 是 1ms，发 100 条就要 100ms。如果 RTT 是 10ms（跨机房），就要 1 秒。

**这太慢了。**

### Pipeline 模式

Pipeline 的思路很简单——**不等签收就继续发**：

```
Leader ──[batch1]──► Follower
Leader ──[batch2]──► Follower  (不等 batch1 的确认)
Leader ──[batch3]──► Follower  (不等 batch2 的确认)
       ◄──[ACK1]────           (batch1 的确认终于到了)
       ◄──[ACK2]────
       ◄──[ACK3]────
```

就像寄快递——你不用等第一个包裹签收了再寄第二个，直接"突突突"连续发。这样网络往返时间就被"重叠"了，吞吐量大幅提升。

### 但 Pipeline 带来了一个棘手的问题：乱序！

网络不保证消息按序到达。你先发 batch1、再发 batch2、再发 batch3，但可能 batch3 的确认先到、batch1 的确认后到。如果你先处理 batch3 的确认（推进 nextIndex），再处理 batch1 的确认——就乱了。

JRaft 怎么解决？**序号机制！**

Replicator 里有三个关键字段：

1. **`reqSeq`**：发送序号。每发一批日志，就分配一个递增序号
2. **`requiredNextSeq`**：期望的下一个确认序号。初始是 0，处理完序号 0 的确认后变成 1
3. **`pendingResponses`**：一个优先级队列，暂存"不是按序到达"的确认

工作流程是这样的：

1. 发 batch1 → seq=0
2. 发 batch2 → seq=1
3. 发 batch3 → seq=2
4. 收到 seq=2 的确认 → `requiredNextSeq` 是 0，不匹配！→ 放入 `pendingResponses` 暂存
5. 收到 seq=0 的确认 → `requiredNextSeq` 是 0，匹配！→ 处理，`requiredNextSeq++` 变成 1
6. 检查 `pendingResponses`：有 seq=2 的，但 `requiredNextSeq` 是 1，不匹配 → 继续等
7. 收到 seq=1 的确认 → 匹配！→ 处理，`requiredNextSeq++` 变成 2
8. 检查 `pendingResponses`：有 seq=2 的，匹配！→ 处理，`requiredNextSeq++` 变成 3

**你看，虽然确认是乱序到达的，但处理顺序始终是 0 → 1 → 2。** 这就是 Pipeline 的有序性保证。

### version——"过期的签收单不要"

还有一个问题：如果 Replicator 因为某些原因被重置了（`resetInflights()`），之前飞出去的 RPC 确认还会回来。这些"旧世界"的确认不能处理——因为 `nextIndex` 已经被重置了。

JRaft 的解决方案是一个 `version` 字段：

1. 每次 `resetInflights()` 时，`version++`
2. 每个 RPC 发出时，闭包里捕获当前 `version`
3. RPC 回调时，检查 `version` 是否匹配。不匹配就丢弃

就像每次"换代"后旧的签收单就作废了——"你这是上一代的签收单，不认。"

## 第四幕：Follower 怎么处理收到的日志？

Leader 通过 Replicator 把日志发给了 Follower，Follower 怎么处理？

### 步骤 1：term 检查

Follower 检查 Leader 发来的 `term` 是不是 >= 自己的 `currTerm`。如果 Leader 的 term 比自己小——说明这是一个"过时的 Leader"（可能是网络分区后恢复的旧 Leader），**拒绝！**

如果 Leader 的 term 比自己大——说明有新 Leader 了，Follower 会 `stepDown(newTerm)`，更新自己的 term。

### 步骤 2：日志匹配检查（prevLogIndex / prevLogTerm）

Leader 发来的每批日志都会附带一个关键信息——"这批日志的前一条日志的 index 和 term 是什么"。

Follower 检查自己本地的日志：我的第 `prevLogIndex` 条日志的 term 是不是 `prevLogTerm`？

- 如果是 → 匹配！说明到 `prevLogIndex` 为止，Follower 和 Leader 的日志是一致的。继续往后追加。
- 如果不是 → 不匹配！说明 Follower 在某个地方和 Leader 的日志出现了分歧。返回失败，Leader 会回退 `nextIndex` 再重试。

这个检查叫做 **Log Matching Property**——它是 Raft 保证日志一致性的关键。原理很简单：

> 如果两个节点在同一个 index 上有相同 term 的日志，那么它们从第 1 条到这条日志的所有内容**完全一致**。

这是一个数学归纳法的结论——因为每条日志在写入时都会检查前一条的一致性（prevLogIndex/prevLogTerm），所以如果第 N 条一致，第 1~N 条也一致。

### 步骤 3：写入本地

检查通过后，Follower 把日志写入自己的 LogManager（内存 + RocksDB），然后回复 Leader："我收到了。"

### 快速回退优化

如果日志不匹配，Leader 需要回退 `nextIndex` 再重试。最朴素的做法是"一次退一条"——但如果 Follower 落后了 1000 条日志，就要回退 1000 次！

JRaft 的优化是：**Follower 在拒绝响应中带上自己的 `lastLogIndex`，Leader 直接跳到 `lastLogIndex + 1`**。一步到位，不需要逐条回退。

## 第五幕：BallotBox——"投票箱"的多数派统计

Leader 收到 Follower 的确认后，怎么知道"超过半数的人都确认了"？这就是 **BallotBox** 的工作。

### 工作原理

把 BallotBox 想象成一排投票箱，**每条日志对应一个投票箱**。

```
投票箱队列：
  [日志101的箱子] [日志102的箱子] [日志103的箱子] [日志104的箱子]
  pendingIndex=101
```

每个投票箱（Ballot）里有一个"还需要多少票"的倒计时器——`quorum`。3 节点集群的 quorum 是 2（3/2+1）。

当 Leader 本地写完日志，算 1 票（`quorum = 2 - 1 = 1`）。
当 Follower B 确认，再算 1 票（`quorum = 1 - 1 = 0`）。
`quorum <= 0`？**提交！**

### commitAt() 的并发设计

多个 Replicator 线程会同时调用 `commitAt()`（因为不同 Follower 的确认可能同时到达）。BallotBox 用 **StampedLock** 来保证线程安全。

为什么用 StampedLock 而不是 `synchronized` 或 `ReentrantLock`？

因为 `getLastCommittedIndex()` 会被**非常频繁**地调用（每次心跳、每次 AppendEntries 都要读它）。StampedLock 支持**乐观读**——先不加锁就读，读完后检查"期间有没有人写过"。如果没有（大部分情况），就省去了加锁的开销。

```java
long stamp = stampedLock.tryOptimisticRead();  // 不加锁
long val = lastCommittedIndex;
if (stampedLock.validate(stamp)) {  // 没人写过？
    return val;                      // 直接返回，零锁开销！
}
// 有人写过，降级为悲观读锁
stamp = stampedLock.readLock();
try {
    return lastCommittedIndex;
} finally {
    stampedLock.unlockRead(stamp);
}
```

### 为什么 `onCommitted()` 在锁外调用？

`commitAt()` 确认多数派后，需要通知 FSMCaller（状态机调用器）。但这个通知 `onCommitted()` 是在**释放写锁之后**才调用的：

```java
} finally {
    stampedLock.unlockWrite(stamp);  // 先释放锁
}
waiter.onCommitted(lastCommittedIndex);  // 再通知
```

为什么？因为 `onCommitted()` 会往 FSMCaller 的 Disruptor 队列里发布事件。如果 Disruptor 队列满了，`publishEvent()` 会阻塞。如果此时还持有 BallotBox 的写锁——其他 Replicator 线程想调 `commitAt()` 就会被阻塞，形成连锁反应，甚至死锁。

**"先释放锁，再做可能阻塞的操作"——这是并发编程的黄金法则。**

### SegmentList 为什么比 ArrayList 好？

BallotBox 的 `pendingMetaQueue` 用的是 `SegmentList` 而不是 `ArrayList`。

日志提交后，投票箱要从头部批量删除。`ArrayList` 的 `removeRange(0, n)` 需要把后面的元素全部往前拷贝——O(n)。

`SegmentList` 把数据分成一段一段的（每段 128 个元素），删除头部时直接丢弃整段——O(n/128)，快了约 128 倍。

## 第六幕：FSMCaller——"传菜窗口"的精妙设计

BallotBox 确认了多数派，接下来要通知状态机执行。但 BallotBox 不能直接调用状态机——原因很多：

1. `commitAt()` 是 Replicator 线程调的，状态机应该在专属线程执行（不能阻塞 Replicator）
2. 多个 Replicator 线程可能同时调 `commitAt()`，如果直接调状态机，状态机就要处理并发——太痛苦了
3. 日志必须按顺序执行，不能先执行 103 再执行 101

FSMCaller 解决了所有这些问题。它的设计核心是一个 **Disruptor 无锁队列 + 单消费者线程**。

### 批量合并——"5 次通知只做 1 次活"

这是 FSMCaller 最精妙的优化。

假设 BallotBox 连续通知了 5 次：

```
onCommitted(101)
onCommitted(102)
onCommitted(103)
onCommitted(104)
onCommitted(105)
```

这 5 个通知都会变成 Disruptor 队列里的 5 个 COMMITTED 事件。Disruptor 的消费者线程会一次取出整个批次（利用 `endOfBatch` 机制）：

- 处理事件 101：`maxCommittedIndex = 101`，不是批次最后一个 → 不处理
- 处理事件 102：`maxCommittedIndex = 102`，不是最后一个 → 不处理
- 处理事件 103：`maxCommittedIndex = 103`，不是最后一个 → 不处理
- 处理事件 104：`maxCommittedIndex = 104`，不是最后一个 → 不处理
- 处理事件 105：`maxCommittedIndex = 105`，**是批次最后一个！** → 调用 `doCommitted(105)`

`doCommitted(105)` 会一次性从 `lastAppliedIndex + 1` 到 105，把所有日志都取出来交给状态机执行。

**5 次 onCommitted() 调用，只触发 1 次 doCommitted()。** 这就是 Disruptor `endOfBatch` 批量合并的威力。

### 三个 Index 的区别

FSMCaller 维护了三个关键 index：

1. **lastCommittedIndex**：BallotBox 通知的"已达多数派"的最大 index
2. **applyingIndex**：当前正在 apply 的 log index（实时更新，用于监控）
3. **lastAppliedIndex**：状态机已成功 apply 的最大 index

正常情况下：`lastAppliedIndex <= applyingIndex <= lastCommittedIndex`

它们的差值代表"积压量"——如果 `lastCommittedIndex` 远大于 `lastAppliedIndex`，说明状态机处理速度跟不上日志提交速度，可能需要优化 `onApply()` 的业务逻辑。

### 用户 onApply() 的正确写法

```java
public void onApply(Iterator iter) {
    while (iter.hasNext()) {
        ByteBuffer data = iter.getData();
        applyToState(data);           // 执行业务逻辑
        if (iter.done() != null) {
            iter.done().run(Status.OK());  // 回调客户端
        }
        iter.next();
    }
}
```

**必须消费完 Iterator 的所有条目！** 如果你提前 return 了，没消费完的日志对应的客户端回调不会被执行——客户端会永远等不到响应（超时）。

## 第七幕：LogManager 的内存与磁盘协作

最后聊一聊 `LogManagerImpl`——它是日志的"总管"，在整个写路径中起到了承上启下的作用。

### 内存先行，异步落盘

Leader 收到写请求后，LogManager 做两件事：

1. **先写内存**：`logsInMemory.addAll(entries)` → 微秒级别，极快
2. **再异步写磁盘**：`diskQueue.publishEvent(done)` → 放到 Disruptor 队列里，由磁盘写线程异步处理

这样 Replicator 可以**立刻从内存读取**刚写入的日志发给 Follower，不需要等磁盘写完。这是一个非常重要的性能优化——**日志复制的延迟不取决于磁盘速度，而取决于网络速度。**

### 内存日志什么时候可以清除？

内存不是无限的，`logsInMemory` 不能无限增长。但清除有两个前提：

1. **已落盘**（`diskId`）：日志已经写到 RocksDB 了，磁盘上有备份
2. **已 apply**（`appliedId`）：状态机已经处理过了，不需要再从内存读

**两个条件都满足才能清除。** JRaft 取 `min(diskId, appliedId)` 作为清除边界。

为什么不是"已落盘就清除"？因为 FSMCaller 的 `IteratorImpl` 可能还要从内存读取日志来 apply（内存读比磁盘读快）。所以要等状态机处理完了才能清除。

### wakeupAllWaiter——通知 Replicator 有新日志

Replicator 发完所有日志后，不能一直轮询"有没有新日志"——太浪费 CPU。它会注册一个回调到 LogManager 的 `waitMap` 里："等有新日志了叫我。"

当 `appendEntries()` 写入新日志后，会调用 `wakeupAllWaiter()`——唤醒所有等待的 Replicator。

**但 `wakeupAllWaiter()` 必须先释放锁再回调。** 为什么？因为回调会触发 Replicator 继续发送日志，Replicator 可能又会调用 `wait()` 注册新的回调——`wait()` 内部需要加写锁。如果 `wakeupAllWaiter()` 持锁回调，就形成了**死锁**。

## 总结：日志复制的"全景图"

让我把整条链路的关键设计决策串起来：

### 性能优化

| 优化 | 原理 | 效果 |
|------|------|------|
| Pipeline 复制 | 不等 ACK 就连续发 | 吞吐量提升 N 倍 |
| 内存先行 | 先写内存，异步落盘 | 复制延迟不取决于磁盘 |
| Disruptor 批量合并 | 多次 onCommitted 合并成一次 doCommitted | 减少状态机调用次数 |
| 乐观读锁 | StampedLock.tryOptimisticRead | 高频读零锁开销 |
| SegmentList | 分段数组，批量删头 O(n/128) | BallotBox 清理速度提升 128 倍 |
| PosHint | 缓存 Follower 在 peers 列表中的位置 | 连续投票 O(1) 命中 |

### 安全保证

| 机制 | 解决的问题 |
|------|-----------|
| prevLogIndex/prevLogTerm | 保证 Leader 和 Follower 的日志一致 |
| reqSeq + requiredNextSeq | Pipeline 下保证确认按序处理 |
| version | 重置后丢弃旧版本的 RPC 确认 |
| quorum + oldQuorum | 配置变更期间的双多数派安全 |
| onCommitted 锁外调用 | 防止 Disruptor 满时死锁 |
| wakeupAllWaiter 锁外回调 | 防止回调中加锁形成死锁 |
| min(diskId, appliedId) 清除 | 保证内存日志不被过早清除 |

### 这四个组件的分工

```
Replicator：负责"发"
    ↓ Follower 确认
BallotBox：负责"数"
    ↓ 多数派确认
FSMCaller：负责"排队 + 合并"
    ↓ 单线程按序
StateMachine：负责"做"
```

每个组件职责单一、接口清晰、通过队列解耦。这是 JRaft 架构设计的一大优点——**高内聚、低耦合**。
