# 用人话聊聊监控：给集群做"体检"

> 这篇文档是 [12-metrics-observability/README.md](./README.md) 的通俗伴读版。

---

## 开场：没有监控的系统就是在裸奔

你去医院体检，医生量血压、测心率、查血常规——这些指标能告诉你身体哪里有问题。

分布式系统也一样。JRaft 集群跑起来后，你怎么知道它"健不健康"？

- 日志复制延迟多少毫秒？→ 延迟飙升说明网络或磁盘有问题
- 每秒处理多少条日志？→ 吞吐量下降可能是 Disruptor 队列堆积
- 选举花了多长时间？→ 选举总超时说明网络不稳定
- Follower 落后 Leader 多少条？→ 落后太多说明 Follower 快扛不住了

**没有这些数据，出了问题你只能猜。有了这些数据，你能秒定位根因。**

---

## 第一幕：三种测量工具

JRaft 用的是 **Dropwizard Metrics**（Java 世界最经典的 metrics 库）。它提供三种基本测量工具：

### 1. Timer（计时器）

"这个操作花了多长时间？"

Timer 不只记录平均值——它记录**分位数**：p50（50% 的请求在这个时间内完成）、p99（99% 的请求在这个时间内完成）、p999（99.9%……）。

为什么分位数比平均值重要？因为平均值会掩盖极端情况。假设 99% 的请求 1ms 完成，1% 的请求 1000ms——平均值是 10.99ms，看起来还行。但 p99=1000ms！那 1% 的请求等了整整 1 秒！

JRaft 的关键路径都有 Timer：
- `fsm-apply-tasks`：状态机应用日志的耗时
- `replicate-inflights-timer`：日志复制的往返耗时
- `handle-read-index`：ReadIndex 处理耗时
- `append-logs`：日志追加耗时

### 2. Counter（计数器）

"这个事件发生了多少次？"

- `read-index-overload-times`：ReadIndex 队列满被拒绝了多少次
- 选举次数、心跳次数……

### 3. Histogram（直方图）

"每批日志有多少条？大小分布是什么样的？"

- `fsm-apply-tasks-count`：每次 onApply 处理了多少条日志
- `handle-read-index-entries`：每次 ReadIndex 请求包含多少条

## 第二幕：信号诊断——"紧急体检按钮"

这是 JRaft 一个非常实用但鲜为人知的功能。

在 Linux 上，你可以给 JRaft 进程发一个 `USR2` 信号：

```bash
kill -USR2 <pid>
```

JRaft 收到信号后，会把**所有 Raft 节点的完整状态** dump 到日志文件里：

```
--- Node: 127.0.0.1:8081/0 ---
state: LEADER
term: 5
lastCommittedIndex: 1234
lastAppliedIndex: 1230
conf: [127.0.0.1:8081, 127.0.0.1:8082, 127.0.0.1:8083]
replicators:
  127.0.0.1:8082: nextIndex=1235, flying=2
  127.0.0.1:8083: nextIndex=1200, flying=0   ← 落后了！
snapshotExecutor: savingSnapshot=false, loadingSnapshot=false
```

**不用重启进程、不用连 JMX、不用任何侵入性操作**——一条命令就能看到所有内部状态。

生产环境救火的时候，这个功能能帮你在 30 秒内定位问题：
- `lastAppliedIndex` 远小于 `lastCommittedIndex` → 状态机处理慢
- 某个 Replicator 的 `nextIndex` 远落后 → 那个 Follower 有问题
- `savingSnapshot=true` 持续很久 → 快照卡住了

## 第三幕：NodeMetrics——"仪表盘"

`NodeMetrics` 是每个 Raft 节点的"仪表盘"。它封装了 Dropwizard 的 `MetricRegistry`，提供两个核心方法：

```java
// 记录延迟（Timer）
nodeMetrics.recordLatency("replicate-inflights-timer", costMs);

// 记录大小（Histogram）
nodeMetrics.recordSize("fsm-apply-tasks-count", taskCount);
```

### 采集指标汇总

| 指标名 | 类型 | 含义 |
|--------|------|------|
| `append-logs` | Timer | 日志追加耗时 |
| `replicate-inflights-timer` | Timer | 日志复制往返耗时 |
| `handle-read-index` | Timer | ReadIndex 处理耗时 |
| `fsm-commit` | Timer | doCommitted 总耗时 |
| `fsm-apply-tasks` | Timer | onApply 业务代码耗时 |
| `read-index` | Timer | ReadIndex 从请求到回调的完整延迟 |
| `fsm-apply-tasks-count` | Histogram | 每次 onApply 处理的日志条数 |
| `handle-read-index-entries` | Histogram | 每次 ReadIndex 请求包含的条目数 |
| `read-index-overload-times` | Counter | ReadIndex 队列满被拒绝次数 |

### 与 Prometheus/Grafana 集成

JRaft 本身不直接对接 Prometheus。你需要自己写一个 `MetricReporter`，定时从 `MetricRegistry` 获取数据，推送到 Prometheus。

或者用 Dropwizard 社区的 `prometheus-metrics` 桥接库，一行配置搞定。

## 第四幕：Disruptor 指标

每个 Disruptor 实例都注册了 `DisruptorMetricSet`，包含：
- `disruptor-buffer-size`：RingBuffer 大小
- `disruptor-remaining-capacity`：剩余容量

如果 `remaining-capacity` 持续接近 0——说明消费者跟不上生产者，队列快满了。这时候你需要：
- 优化消费者（如 `onApply` 太慢）
- 或者增大 RingBuffer 大小（`raftOptions.setDisruptorBufferSize()`）

---

## 总结

JRaft 的可观测性设计：

| 设计 | 解决的问题 |
|------|-----------|
| Dropwizard Metrics | 标准化的 Timer/Counter/Histogram 采集 |
| 分位数统计 | 发现 p99/p999 延迟尖刺 |
| Unix Signal 诊断 | 零侵入式实时状态 dump |
| Disruptor 指标 | 监控队列积压情况 |
| NodeMetrics 封装 | 统一的指标采集 API |

**一句话总结**：监控不是锦上添花——它是生产环境的"眼睛"。没有它，你就是蒙着眼睛开飞机。
