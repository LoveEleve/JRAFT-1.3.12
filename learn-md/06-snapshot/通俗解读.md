# 用人话聊聊快照：RPG 游戏存档的艺术

> 这篇文档是 [06-snapshot/README.md](./README.md) 的通俗伴读版。README 里有完整的保存/安装时序图、分支穷举和引用计数源码分析，这篇文档用讲故事的方式帮你**理解快照为什么存在、每一步为什么这样做**。

---

## 开场：如果日志无限增长……

大家好，前面我们花了很多篇幅聊日志——写日志、存日志、复制日志。但有一个问题一直没提：

**日志会无限增长吗？**

答案是：**如果不做处理，是的。** 每条写请求都会生成一条日志，一天几百万条，一周几千万条。用不了多久，磁盘就满了。

而且，日志太长还会导致另外两个严重问题：

1. **重启太慢**：节点重启后，需要重放所有日志来恢复状态机——日志有一千万条，就要重放一千万条。这可能需要几十分钟甚至几小时
2. **新手入门太慢**：新节点加入集群，需要从头复制所有日志——Leader 得把一千万条日志全发过去

**快照（Snapshot）就是解决这三个问题的。**

---

## 第一幕：快照就是"游戏存档"

打过 RPG 游戏吧？你打了 3 个小时，角色升到 50 级，有了屠龙刀，做了 100 个任务。这时你存了个档。

存档里记的是什么？**不是你按了哪些键**（那是"操作日志"），而是**你当前的完整状态**——50 级、HP 500、攻击力 200、背包里有屠龙刀……

存档之后，前 3 个小时的操作记录就可以删了——反正你有存档了，不需要"从头开始重玩一遍"来恢复到这个状态。

Raft 的快照完全一样：

- **日志 = 操作记录**（"put key=foo value=bar"、"delete key=baz"……）
- **快照 = 存档**（状态机的当前完整状态）
- **存档后旧日志可以删除** = `truncatePrefix`

所以快照的核心用途就是三个字：**省空间、快恢复、帮新手。**

## 第二幕：快照保存——"什么时候该存档？"

### 定时触发

JRaft 有一个 `snapshotTimer`，每隔 `snapshotIntervalSecs` 秒触发一次："该存档了吗？"

但不是每次触发都会真的存档。JRaft 会做几个检查：

1. **有新内容吗？** 如果上次存档后一条新日志都没有（`lastAppliedIndex == lastSnapshotIndex`），存档毫无意义——存出来的内容和上次一模一样
2. **距离够远吗？** `snapshotLogIndexMargin` 配置了"至少新增多少条日志才做快照"。如果只新增了 3 条，做一次快照的成本太大了（可能要写几 GB 数据），不值得
3. **有人在存档吗？** 同一时刻只能有一个快照操作——要么在"存档"（save），要么在"读档"（install），不能同时进行

### 为什么同一时刻只能一个快照操作？

想象两个人同时往同一个存档文件里写数据——写出来的东西肯定是乱的。快照操作涉及大量文件 IO，并发执行会导致文件损坏。

JRaft 用两个 `volatile boolean` 来实现互斥：`savingSnapshot` 和 `loadingSnapshot`。任何快照操作开始前，都会检查这两个标志。

### 存档的具体步骤

存档分几步：

1. **写入临时目录**：快照数据先写到 `temp` 目录里，不直接写到正式的 `snapshot_N` 目录
2. **用户自定义保存**：框架不知道你的状态机里有什么数据——它调用 `StateMachine.onSnapshotSave(writer, done)`，让你自己把数据写出去。你可以写成任何格式：JSON、Protobuf、直接 dump 数据库……写完后调用 `done.run(Status.OK())`
3. **保存元数据**：框架自动保存一个叫 `__raft_snapshot_meta` 的文件，记录这次快照包含到哪条日志（`lastIncludedIndex/Term`）、当前集群配置（`peers`）
4. **原子重命名**：`temp` 目录重命名为 `snapshot_N`——这是**原子操作**，要么完整成功，要么不变。不会出现"写了一半"的快照目录

### 为什么要用临时目录？

假设直接写入正式目录。写到一半崩溃了——正式目录里的快照是半成品，无法使用。而旧的快照已经被覆盖了——数据丢了！

用临时目录就没这个问题：写到一半崩溃，`temp` 目录里的半成品会在下次启动时被清理掉，旧的 `snapshot_N` 目录还好好的。

## 第三幕：快照安装——"给新手发存档"

快照安装是 Raft 里最复杂的流程之一。我们分"为什么需要"和"怎么做"来聊。

### 为什么需要安装快照？

两种场景：

**场景 1：新节点加入**

假设集群已经运行了一年，有 1000 万条日志，做了快照后只保留了最近 10 万条。新节点加入时，Leader 想从第 1 条开始发日志——但第 1 到第 990 万条已经被删了！

怎么办？Leader 把最新的快照发给新节点："这是 990 万条日志执行后的状态，你直接加载这个快照，然后从第 990 万零 1 条开始追日志。"

**场景 2：Follower 掉线太久**

Follower 掉线了好几天，Leader 的日志早就被截断了。Replicator 发现 Follower 需要的日志已经不存在了——切换到 Snapshot 模式，发快照。

### 安装的具体步骤

安装比保存复杂得多，因为涉及**网络传输**：

1. **Leader 发起 InstallSnapshot RPC**：告诉 Follower "我要给你发个快照，URI 是这个"
2. **Follower 创建 Copier**：Copier 是一个"下载器"，负责从 Leader 下载快照文件
3. **先下载元数据**：`__raft_snapshot_meta`——了解快照包含哪些文件、到哪条日志
4. **过滤已有文件**：如果 Follower 本地已经有某些文件（上次下载到一半崩溃了），可以跳过——**断点续传**！
5. **逐个下载文件**：每个文件创建一个 Session，分块下载
6. **原子替换**：下载完毕，`temp` → `snapshot_N`
7. **加载到状态机**：调用 `StateMachine.onSnapshotLoad(reader)`——用户从快照恢复状态机
8. **截断旧日志**：`logManager.setSnapshot(meta)`——快照之前的日志全删了

### 文件传输为什么不用 RPC？

快照可能很大——几个 GB 甚至几十 GB。如果把整个快照塞进一个 RPC 请求里，网络缓冲区和内存都会爆。

JRaft 的做法是 `FileService`——Leader 端启动一个"文件服务"，Follower 像从网盘下载文件一样，一块一块地拉。每次拉一个 chunk（默认 128KB），拉完一块再拉下一块。

这就像你从百度网盘下载一个 5GB 的文件——浏览器不会一次性把 5GB 都加载到内存里，而是分块下载、分块写入磁盘。

## 第四幕：引用计数——"别人还在看的存档不能删"

这是一个精巧但容易被忽略的设计。

假设当前有两个快照：`snapshot_100`（旧的）和 `snapshot_200`（新的）。正常情况下，`snapshot_100` 应该被删掉。

但如果此时有一个 Follower 正在从 Leader 的 `snapshot_100` 下载文件呢？Follower 正在下载，你把文件删了——**下载失败！**

JRaft 用**引用计数（Reference Counting）**解决这个问题：

- `ref(index)`：有人要读这个快照 → 引用计数 +1
- `unref(index)`：读完了 → 引用计数 -1
- 引用计数减到 0 → 可以安全删除

具体来说：

```
open(snapshot_100) → ref(100)：计数 0→1
                                    此时不会被删除
close(reader)     → unref(100)：计数 1→0 → 自动删除！
```

这和操作系统管理内存的思路一模一样——只要还有指针指向这块内存，就不能释放。

### 引用计数的并发安全

引用计数用的是 `ConcurrentHashMap<Long, AtomicInteger>`——无锁并发安全。`AtomicInteger.incrementAndGet()` 和 `AtomicInteger.decrementAndGet()` 都是原子操作，多个线程同时操作也不会出问题。

## 第五幕：那些容易踩的坑

### 坑 1：忘记调用 `done.run()`

用户实现 `onSnapshotSave(writer, done)` 时，写完数据后**必须**调用 `done.run(Status.OK())`。如果忘了——`savingSnapshot` 永远是 `true`，后续所有快照请求都会被拒绝（返回 EBUSY），节点的日志会无限增长，最终磁盘爆满。

这是生产环境中最常见的快照相关 bug。

### 坑 2：快照文件太大，传输时间超过选举超时

快照几十 GB，传输需要好几分钟。但选举超时只有几秒。Follower 在安装快照期间收不到正常的 AppendEntries 心跳，可能触发选举。

解决方案：配置 `snapshotThrottle` 限制快照传输带宽（给正常心跳留余量），同时适当增大 `electionTimeoutMs`。

### 坑 3：快照和日志的"交接"出错

快照安装后，`logManager.setSnapshot(meta)` 会截断旧日志。如果快照的 `lastIncludedIndex` 和日志的 `firstLogIndex` 之间有"空洞"——比如快照到 100，但日志从 150 开始——中间 101-149 的日志就丢了。

JRaft 的保护机制是：`setSnapshot()` 内部会检查并处理这些 edge case，但用户需要保证自己的 `onSnapshotLoad` 实现是正确的——加载快照后，状态机的状态确实代表了 `lastIncludedIndex` 时刻的完整状态。

## 第六幕：互斥与并发——快照操作的"交通规则"

快照涉及的并发场景很多，JRaft 有一套严格的"交通规则"：

### 规则 1：保存和安装互斥

正在保存快照时，来了一个安装请求 → **拒绝安装**（返回 EBUSY）
正在安装快照时，来了一个保存请求 → **拒绝保存**（返回 EBUSY）

### 规则 2：新的安装请求可以取消旧的下载

如果 Follower 正在下载 `snapshot_100`，这时候 Leader 发来了 `snapshot_200` 的安装请求——`snapshot_200` 更新，应该安装它。JRaft 会取消旧的下载（`curCopier.cancel()`），然后返回 EBUSY 让 Leader 重试。

**为什么不直接开始新下载？** 因为 `cancel()` 是异步的，旧 copier 可能还在写临时目录。如果同时开始新下载，两个 copier 会冲突。返回 EBUSY 后，Leader 下次心跳时会重发 InstallSnapshot，此时旧 copier 已经退出了。

### 规则 3：正在加载的快照不能被中断

下载完毕后，`loadingSnapshot=true`，正在调用 `StateMachine.onSnapshotLoad(reader)`。这时候来了更高 term 的消息，`interruptDownloadingSnapshots(newTerm)` 被调用。

它会检查 `loadingSnapshot`——如果正在加载，**不中断**。因为状态机正在"重建"自己的状态，中断会导致状态机处于不一致状态——半建不建的，比没快照还危险。

### 规则 4：`logManager.setSnapshot()` 必须在锁外调用

`setSnapshot()` 会触发日志截断和 Replicator 唤醒。如果在 `SnapshotExecutorImpl.lock` 内调用，可能和 Replicator 的锁产生死锁。

JRaft 用 `doUnlock` 标志位来处理：先释放锁，调用 `setSnapshot()`，再重新加锁。这个模式在 JRaft 里非常常见（前面选举和日志复制里也见过）。

---

## 总结

快照机制的核心设计：

| 设计 | 解决的问题 |
|------|-----------|
| 定期快照 | 控制日志大小、加速重启 |
| 快照安装 | 新节点快速加入、落后节点追赶 |
| 临时目录 + 原子重命名 | 崩溃安全：不会出现半成品快照 |
| 引用计数 | 被读取的快照不会被删除 |
| 保存/安装互斥 | 避免并发写文件导致数据损坏 |
| 断点续传 | 大快照传输中崩溃后不需要从头开始 |
| FileService 分块传输 | 大快照不会撑爆内存 |
| `doUnlock` 模式 | 防止锁嵌套导致死锁 |

**一句话总结**：快照 = **定期给状态机拍照，扔掉旧日志，新手直接拿照片恢复**。核心挑战是大文件传输、并发互斥和崩溃安全。
