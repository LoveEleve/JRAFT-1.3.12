# Source Code Reading Skills & Rules（通用版，共 16 条）

---

## 1. 数据结构优先

程序 = 数据结构 + 算法，先搞懂数据结构再看行为。
每进入新模块，先梳理核心类的字段、继承关系、持有关系，画出对象关系图后再看方法逻辑。

> **补充**：先看字段的类型和初始化时机。字段是 `final` 还是 `volatile` 往往直接揭示并发设计意图。

---

## 2. 问题导向阅读

带着问题读源码，不做无目的的逐行扫描。
每次阅读前先明确要回答的问题，读完后必须能用自己的话回答这个问题。

> **补充**：使用以下问题清单模板：
> - 这个模块解决什么问题？
> - 入口在哪里？
> - 核心状态在哪里维护？
> - 异常/边界情况怎么处理？

---

## 3. 分层阅读法

先接口后实现，先黑盒后白盒。
第一层 API 使用者视角 → 第二层架构师视角 → 第三层实现者视角，不跳级，每层都要过。

> **补充**：第一层可以明确为"先看单元测试"。测试用例是最好的 API 使用文档，往往比 Javadoc 更直接。

---

## 4. 请求生命周期为主线

以请求生命周期为主线，而非按模块逐个啃。
始终围绕"一个请求从进来到出去的完整路径"来组织阅读。

> **补充**：对于非请求型框架（如调度器、存储引擎），替换为"以一个核心操作的完整路径为主线"。
> 例如 sofa-jraft 中的 `applyTask → 日志复制 → 状态机应用` 这条链路。

---

## 5. 对比思维贯穿始终

横向对比 + 纵向对比 + 跨框架对比。
- **横向对比**：不同实现方案之间
- **纵向对比**：git history 演进
- **跨框架对比**：相似框架的设计差异

> **补充**：加入"版本对比"——看 CHANGELOG 或 Release Notes，了解某个设计是在哪个版本引入的、解决了什么历史问题，比单纯看 git history 更高效。

---

## 6. Debug 是最终裁判

对执行路径有疑问时必须通过 Debug 验证，不能靠猜。

> **补充**：加入"写单元测试验证你的理解"。Debug 是被动验证，写测试是主动验证，后者对理解的加深更彻底。

---

## 7. 四类图输出

每个模块产出四类图（按优先级排序）：

1. **对象关系图**（最先画，是后三类图的基础）：核心类之间的持有/继承关系
2. **时序图**（理解主流程）：方法调用的时间顺序
3. **线程交互图**（理解并发模型）：哪个线程做什么事，线程间如何交互
4. **状态机图**（理解生命周期，并非所有模块都有）：状态流转与触发条件

---

## 8. 抓住不变式（Invariant）

以不变式作为理解的锚点。
每个模块总结 2-3 个核心不变式（即在任何时刻都必须成立的规则/约束）。

> **补充**：不变式总结后，进一步追问"这个不变式是如何被保证的？"（锁？CAS？单线程？协议约束？）
> 这会直接引出并发设计的核心。

---

## 9. AI 解释源码格式

结构化分析模板：
1. 先说解决什么问题
2. 核心数据结构
3. 关键流程（配调用链 / 时序图）
4. 为什么这样设计（设计动机、trade-off）
5. 涉及面试高频考点标注 📌
6. 涉及生产踩坑标注 ⚠️

---

## 10. 全量分析，分批拆解

源码必须全量分析，但可以分批拆解：
- 每个方法/类的源码都要完整分析，不跳过任何一行
- 但一次只聚焦一个方法或逻辑块
- 每段源码必须配解释
- 分析完后用"为什么不用 xxx 方案"深化理解
- **最终目标**：模块内每一行源码都被分析过，无死角

> **补充**：建立"源码阅读进度表"——记录哪些类/方法已分析、哪些待分析，防止遗漏，也方便复盘。

---

## 11. 始终关联生产实践

讲原理时关联生产常见问题和最佳实践。
- 标注"面试常考" 📌
- 标注"生产踩坑" ⚠️

---

## 12. 渐进式深入

不一次性信息过载：
- 一次只讲一个核心概念，讲透再进入下一个
- 遇到复杂机制先给简化模型，再逐步补充细节

> **补充**：先建立简化心智模型，再用真实细节修正它。允许自己先有一个"不完全正确但可用"的模型，再迭代修正，比追求一次性完全正确更高效。

---

## 13. 父类构造链必查 ⭐

分析任何对象的创建过程时，必须沿着继承链向上追溯所有父类的构造函数：
- 逐层列出每个父类构造中创建/初始化的字段和对象
- 绝不能只看当前类的构造函数就结束
- 父类构造中往往隐藏着关键的初始化逻辑
- 这是最容易遗漏的盲区

> **补充**：扩展为"构造链 + 静态初始化块 + 懒加载字段"三位一体。
> 很多框架的关键初始化逻辑藏在静态块或第一次访问时的懒加载中（如 Netty 的 `InternalThreadLocalMap`）。

---

## 14. 异常与降级路径必读 ⭐

不只读 happy path，必须追踪异常处理、超时、重试、降级的完整路径。

生产故障 90% 发生在异常路径上，而大多数人只读了正常路径。

对每个 `catch` 块必须标注：
- 是**吞掉**了？
- 是**重试**了？
- 是**向上抛**了？
- 是**触发了降级**？

> ⚠️ **生产踩坑**：异常被静默吞掉是线上问题最难排查的根因之一。

---

## 15. 所有图使用 Mermaid 绘制

第 7 条中要求输出的四类图（对象关系图、时序图、线程交互图、状态机图），以及任何分析过程中产出的图，**一律使用 Mermaid 语法绘制**，不使用 ASCII 图或其他格式。

> **原因**：Mermaid 图可直接在 Markdown 中渲染，版本可控、易于修改、阅读清晰，且可与源码分析文档无缝集成。

**各类图对应的 Mermaid 图类型参考：**

| 图类型 | Mermaid 语法 |
|--------|-------------|
| 对象关系图 | `classDiagram` |
| 时序图 | `sequenceDiagram` |
| 线程交互图 | `sequenceDiagram`（以线程为参与者） |
| 状态机图 | `stateDiagram-v2` |
| 流程图 / 调用链 | `flowchart TD` |

---

## 16. 一切分析与结论必须基于本地源码 ⭐

所有对源码行为、字段名、方法名、类结构、调用链的描述，**必须通过实际查阅本地源码来确认**，不允许凭印象、凭经验或凭文档推断后直接输出结论。

**执行要求：**
- 提到任何类名、方法名、字段名之前，必须先在本地源码中搜索确认其存在
- 描述调用链或执行路径之前，必须追踪源码中的实际调用关系
- 描述状态、枚举值、配置项之前，必须查看对应的源码定义
- 如果本地源码中找不到对应内容，必须明确说明
