## 开场：为什么选举是 Raft 最有趣的部分？

大家好，上一篇我们聊了节点启动，最后一步是启动 `electionTimer`——"选举超时定时器"。定时器启动后，如果在规定时间内没有收到 Leader 的心跳，节点就会触发选举。

**选举是 Raft 里最"戏剧性"的部分**——有竞选、有投票、有拉票、有退出、有策略、有意外。如果你曾经参加过学生会竞选或者看过真人秀投票节目，你会发现 Raft 的选举和现实生活中的选举惊人地相似。

但和现实选举不同的是，Raft 的选举必须满足一个**铁律**：

> **任何时刻，整个集群最多只能有一个 Leader。如果有两个 Leader 同时存在，数据就可能不一致——这比没有 Leader 还可怕。**

所以 Raft 的选举规则是围绕这个铁律精心设计的。让我们从头开始，一步步理解。

---

## 第一幕：一切从超时开始

假设集群里有三个节点：A、B、C。A 是 Leader，B 和 C 是 Follower。

正常情况下，A 每隔一小段时间就给 B 和 C 发一条心跳——"我还在呢！"。B 和 C 收到心跳后，就知道 Leader 还活着，继续安心当 Follower。

但有一天，A 所在的服务器突然宕机了。

B 等了一会儿——没收到心跳。"奇怪，A 怎么不说话了？"

B 又等了一会儿——还是没收到心跳。B 心里开始嘀咕了——"A 不会挂了吧？"

当 B 等待的时间超过了 `electionTimeoutMs`（选举超时时间），B 做出了一个决定——**"我要出来竞选了！"**

**但是，B 不会直接竞选。** 它会先做一个很聪明的事情——**PreVote（预投票）。**

---

## 第二幕：PreVote——"先在小群里吹吹风"

### 为什么需要 PreVote？

在解释 PreVote 之前，让我先讲一个**没有 PreVote 会出什么问题**的故事。

假设集群有三个节点：A（Leader）、B（Follower）、C（Follower）。现在 C 的网络出了问题——C 和 A、B 之间的网络全断了，但 A 和 B 之间正常。

C 收不到 A 的心跳了，等超时后，C 觉得"Leader 挂了"，开始竞选。竞选时要做一件事：**`currTerm++`**——把自己的任期号加 1。

但 C 是一个人在竞选啊！它给 A 和 B 发投票请求，网络不通，都收不到。等投票超时，C 再竞选一次——`currTerm++`，又加 1。就这样一直循环，C 的 `currTerm` 越涨越高……从 5 涨到 10、涨到 50、涨到 100……

过了一段时间，C 的网络恢复了。C 的 `currTerm` 已经是 100 了，而 A 的 `currTerm` 还是 5（A 一直在正常工作，没有选举过）。

这时候 C 给 A 发了一条消息，A 一看——"天哪，对方的 term 是 100，比我的 5 大多了！按照 Raft 的规则，我必须退位（stepDown）！"

于是 A 被迫退位，集群短暂没有 Leader。然后重新选举，选出来的可能不是最优的。

**这就是"无效选举污染 term"的问题。** C 明明是自己网络断了，它的竞选完全没有意义，但它的超高 term 却像病毒一样，一恢复网络就"感染"了正常工作的集群。

### PreVote 怎么解决这个问题？

PreVote 的思路很简单——**竞选之前，先偷偷问一圈："大家觉得我有资格竞选吗？"**

具体来说：

1. C 超时了，想竞选
2. C **不急着增加 term**，而是先给 A 和 B 发一个"预投票请求"："如果我要竞选，你们会支持我吗？我的 term 将会是 6（currTerm+1），我的日志是最新的到第 X 条。"
3. A 和 B 收到请求后检查：
   - "我的 Leader 还活着吗？"——A 是 Leader，当然活着。B 刚收到 A 的心跳，也认为 Leader 还活着
   - 既然 Leader 还活着，就没必要选举→ **拒绝！**
4. C 发现没有拿到多数票（0 票），竞选取消
5. **关键：C 的 term 没有增加！** 因为 PreVote 只是"问一问"，不是正式竞选

现在我们把场景换成 A 真的挂了：

1. C 超时，发起 PreVote
2. C 给 A 发预投票请求——A 没回应（挂了）
3. C 给 B 发预投票请求——B 检查："我的 Leader 还活着吗？上次收到心跳是很久以前了，超时了……Leader 可能挂了。C 的日志够新吗？够新。"→ **同意！**
4. C 拿到了 B 的同意（加上自己算一票，2/3 多数派）→ PreVote 通过！
5. 现在 C 才正式 `currTerm++`，进入正式竞选

**你看，PreVote 就像一道"过滤器"——把那些"自以为 Leader 挂了但其实只是自己网络断了"的无效竞选给拦住了。**

### PreVote 的 term 为什么用 `currTerm+1` 而不是 `currTerm`？

这是一个很精妙的设计。C 发 PreVote 请求时，用的 term 是 `currTerm + 1`——"如果我正式竞选，我的 term 将会是这个值"。

为什么？因为对方需要拿这个 term 来做判断。如果 C 用的是 `currTerm`（和对方一样大），对方就没法判断"C 是不是有更新的信息"。用 `currTerm + 1`，让对方知道"如果我正式选举，我会有这么大的 term"，这样对方才能正确评估。

**但注意：C 本地的 `currTerm` 并没有真的加 1。** 只有 PreVote 通过了，正式进入 `electSelf()` 时，才会真正 `this.currTerm++`。

---

## 第三幕：electSelf——"正式竞选"

PreVote 通过了，说明大家认可了"确实需要一次选举"。现在 C 要正式开始竞选了。

### 第一步：C 给自己投票

C 做了几件事：
1. 停掉 `electionTimer`（已经在竞选了，不需要再触发超时）
2. 状态从 `FOLLOWER` 变为 `CANDIDATE`
3. **`currTerm++`**——这是真正增加 term 的时刻！
4. 给自己投一票：`votedId = self`
5. 启动 `voteTimer`——"如果选举超时了还没出结果，就重新来"

### 第二步：拉票

C 给集群里的所有其他节点发 `RequestVote` RPC——"大家好，我是 C，我的 term 是 6，我的日志到第 100 条。请投我一票！"

### 第三步：持久化投票记录

C 调用 `metaStorage.setTermAndVotedFor(6, C)` 把 "term=6，投给了 C" 写到磁盘上。

**为什么要写磁盘？** 因为如果 C 投完票就宕机了，重启后如果忘记了自己投过票，可能会再投一次——给别人。这样同一个 term 里就有两张票了，可能导致两个节点同时获得多数票，出现两个 Leader——**脑裂！**

所以 `votedFor` 必须持久化。宕机重启后，从磁盘恢复 `votedFor`，就知道自己已经投过了。

### 第四步：等结果

C 发出去的投票请求，一个个回来了。让我们看看其他节点是怎么处理的。

---

## 第四幕：投票——"要不要投给它？"

B 收到了 C 的投票请求：`RequestVote(term=6, candidateId=C, lastLogIndex=100, lastLogTerm=5)`

B 需要做以下几个检查：

### 检查 1：term 比较

B 看了一下自己的 term——是 5。C 的 term 是 6，比自己大。Raft 有一个铁规矩：**看到比自己大的 term，必须立刻 stepDown（退位/更新到新 term）。**

所以 B 先把自己的 term 更新为 6，状态设为 Follower（如果不是的话），清空 `votedFor`（新 term 里还没投过票）。

### 检查 2：日志新旧比较

B 检查 C 的日志是不是比自己新（或者一样新）。比较规则是：
- **先比 term**：C 最后一条日志的 term 是 5，B 自己最后一条日志的 term 也是 5——一样
- **term 一样的话再比 index**：C 的 lastLogIndex 是 100，B 的也是 100——一样

C 的日志 >= B 的日志，检查通过。

**为什么要比较日志新旧？** 因为 Leader 需要有集群里"最新的"日志。如果选了一个日志落后的节点当 Leader，它上任后会覆盖掉别人更新的日志——那些日志可能已经被提交了——**数据就丢了！**

所以投票规则是：**只有日志至少和我一样新的候选人，我才投票给它。**

### 检查 3：这个 term 里我投过票了吗？

B 检查自己的 `votedFor`——刚才 stepDown 时清空了，还没投过。OK，可以投。

如果 B 已经在 term=6 里投给了另一个人（比如 D），那即使 C 的日志更新，B 也不会再投给 C——**同一个 term 只能投一票。** 这是 Raft 安全性的核心保证。

### 投票！

所有检查都通过了，B 决定投给 C。B 做两件事：
1. `votedFor = C`，写到磁盘上
2. 返回 `granted=true` 给 C

同时，B 调用了 `stepDown()`——这会重置 B 的 `electionTimer`。为什么？**因为 B 刚投了票，说明它认可了这次选举。如果 B 不重置选举定时器，B 自己也可能很快超时发起选举，和 C 的选举打架——选票分裂。**

---

## 第五幕：统计票数——"我赢了吗？"

C 陆续收到了回复。集群是 3 个节点（A、B、C）：

- C 自己投了自己 → 1 票
- B 回复 `granted=true` → 2 票
- A 没回应（宕机了）→ 0 票

2 票 / 3 总 = 多数派！**C 赢了！**

但如果 C 收到了一个回复说"我的 term 是 10（比你大）"——C 会立刻 `stepDown(10)`，放弃竞选。因为有人的信息比自己新，说明自己可能"过时了"。

### Ballot 是怎么计票的？

JRaft 里用 `Ballot` 来计票。它的设计很简单但很巧妙：

1. 初始化时，`quorum = peers.size()/2 + 1`（3 个节点，quorum = 2）
2. 每收到一票 `granted=true`，`quorum--`
3. `quorum <= 0` 时，`isGranted() = true`

为什么是"倒计时"而不是"正计数"？因为倒计时只需要检查 `<= 0`，比正计数（`count >= quorum`）少一次比较。在 Raft 这种性能敏感的场景，每一点优化都值得。

而且 Ballot 支持**联合共识（Joint Consensus）**——配置变更期间，需要同时获得新旧两个配置的多数票。`Ballot` 内部维护了两组计数器：`quorum` 和 `oldQuorum`，`isGranted()` 要求两者都 <= 0。这个我们后面讲配置变更的时候再细说。

---

## 第六幕：becomeLeader——"上任第一天"

C 赢了选举，调用 `becomeLeader()`。上任第一天，C 要做好几件事：

### 1. 停掉投票定时器

选举已经结束了，`voteTimer` 可以停了。

### 2. 状态变更

`state = LEADER`，`leaderId = self`。

### 3. 给每个 Follower 派一个"抄写员"

还记得上一篇提到的 `Replicator` 吗？Leader 会为每个 Follower 创建一个 Replicator——"你去负责把日志抄给 Follower B"。

### 4. 提交一条空日志

这是一个容易被忽略但**极其重要**的步骤。

Raft 论文有一个规则：**Leader 不能直接提交前任 Leader 遗留的日志。** 为什么？让我讲一个著名的 corner case（Raft 论文的 Figure 8）：

假设前任 Leader A 在 term=2 时写了一条日志（index=3），只复制给了 B，还没来得及提交就挂了。现在 C 当选 Leader（term=3），C 发现 index=3 这条 term=2 的日志在多数派节点上了——能不能直接标记它为"已提交"？

**不能！** 因为如果 C 提交了这条 term=2 的日志，然后 C 也挂了，接下来选出来的新 Leader 可能是一个**没有这条日志的节点**（它可能有一条 term=3 的不同日志覆盖了 index=3 的位置）。这就导致"已提交的日志被覆盖了"——违反 Raft 的安全性。

解决方案是：**Leader 当选后，先写一条自己 term 的空日志（no-op）。这条空日志被多数派确认后，它之前所有的日志（包括前任遗留的）都间接被提交了。**

在 JRaft 中，`becomeLeader()` 调用 `confCtx.flush()` 来实现这个 no-op 日志。

### 5. 启动 stepDown 定时器

Leader 需要定期检查——"我还能联系到多数派吗？"如果网络分区了，Leader 发现自己和大多数节点失联了，就应该主动退位，让其他分区的节点选出新 Leader。

`stepDownTimer` 的间隔是 `electionTimeoutMs / 2`——比 Follower 的选举超时快一倍。这保证了 Leader 能在 Follower 超时之前就检测到问题并退位。

## 番外篇一：选举超时为什么要随机化？

还记得 `init()` 里创建 `electionTimer` 时有一个 `adjustTimeout` 方法吗？它会把超时时间在 `[electionTimeoutMs, electionTimeoutMs + maxElectionDelayMs)` 范围内随机化。

**为什么要随机？**

假设不随机——所有节点的超时时间都是精确的 1000ms。Leader 挂了后：

1. 1000ms 时，A、B、C **同时**超时
2. A、B、C **同时**给自己投票
3. A 投自己 1 票，B 投自己 1 票，C 投自己 1 票
4. 没有人拿到 2 票（多数派）
5. 所有人竞选失败
6. 等下一轮超时——**又是同时超时，又是同时投自己，又是全部失败**

这就是**选票分裂（split vote）**——理论上可以无限循环，集群永远选不出 Leader。

随机化打破了这种同步：A 超时 900ms、B 超时 1100ms、C 超时 1300ms。A 最先超时，在 B 和 C 还没超时的时候就发出了投票请求。B 和 C 收到请求后，因为还没自己竞选，所以会投给 A。A 轻松拿到多数票。

**随机化是 Raft 保证选举终将成功（liveness）的关键机制。**

---

## 番外篇二：优先级选举——"让最优秀的人当 Leader"

有时候你希望某台特定的服务器当 Leader——比如它的硬件配置最好，或者它离客户端最近。JRaft 支持给每个节点设置优先级（`priority`）。

但优先级选举不是"高优先级直接当选"——那太霸道了。它的机制更加优雅：

### 规则

1. `priority = 0`：永远不参与选举（"我永远是观众"）
2. `priority = -1`：禁用优先级，按普通规则参与
3. `priority > 0`：数字越大，优先级越高

### 工作方式

集群启动时，每个节点会知道"集群中最高优先级是多少"——这叫 `targetPriority`。

当选举超时触发时，低优先级节点会检查：**"我的优先级比 targetPriority 低吗？"如果是，跳过本轮选举，把机会让给高优先级节点。**

但这样有一个问题：**如果高优先级节点挂了呢？** 低优先级节点一直"让贤"，谁也不选举，集群就瘫痪了。

JRaft 的解决方案是**优先级衰减（decay）**：

- 低优先级节点第一次超时：跳过，让贤
- 低优先级节点第二次超时：`targetPriority` 衰减（降低门槛）
- 衰减公式：`targetPriority -= max(decayPriorityGap, targetPriority / 5)`
- 持续衰减，直到 `targetPriority` 降低到低于自己的优先级——这时低优先级节点终于可以参与选举了

**这就像排队买票：VIP 优先，但如果 VIP 迟迟不来，等了两轮之后，普通客户也可以上了。**

## 番外篇三：stepDown——Raft 里最"通用"的方法

`stepDown()` 在 JRaft 里被很多地方调用，它做的事情可以概括为"一键重置为 Follower"：

1. 如果是 Candidate → 停掉 `voteTimer`
2. 如果是 Leader → 停掉 `stepDownTimer`，清空 `BallotBox`，通知状态机 `onLeaderStop`
3. 状态设为 `FOLLOWER`
4. 清空 `leaderId`
5. 更新 `lastLeaderTimestamp`（重置选举超时的起点）
6. 如果 `term > currTerm` → 更新 `currTerm` 并清空 `votedFor`（新 term 里还没投过票）
7. 重启 `electionTimer`

你看，不管你之前是什么状态，`stepDown()` 都能把你"一键归零"回 Follower。这就是为什么 `init()` 里也调用了 `stepDown()`——它不只是"退位"，更是一个通用的"状态初始化/重置"方法。

## 番外篇四：Leader Transfer——"主动让位"

有时候你想主动把 Leader 从 A 转移到 B（比如要对 A 做维护）。这叫 **Leader Transfer**。

流程是：

1. A（当前 Leader）调用 `transferLeadershipTo(B)`
2. A 检查 B 的日志是否和自己一样新。如果不是，先等 Replicator 把日志追上
3. B 的日志追上后，A 给 B 发 `TimeoutNow` 消息——"你现在立刻发起选举！"
4. B 收到后，**跳过 PreVote**，直接 `electSelf()`——因为这是 Leader 主动让位，不需要再"试探"
5. B 竞选成功，成为新 Leader
6. A 收到 B 的心跳（或更高 term 的消息），自动 `stepDown()`

**为什么 B 可以跳过 PreVote？** 因为 PreVote 的目的是防止"网络隔离的节点无效竞选"。但这里是 Leader 主动发起的转移，Leader 已经确认了 B 的日志是最新的——没有必要再"试探"。

## 第七幕：那些容易被忽略的并发问题

选举过程涉及大量并发操作——超时定时器在一个线程、RPC 回调在另一个线程、用户调用在第三个线程。JRaft 怎么保证这些操作不打架？

### ABA 防御模式

在 `preVote()` 和 `electSelf()` 里，有一个很精妙的"ABA 防御"模式：

```
writeLock.lock()
oldTerm = this.currTerm
writeLock.unlock()          ← 释放锁，因为下面要做 IO
lastLogId = getLastLogId()  ← 这一步可能涉及磁盘读，不能持锁
writeLock.lock()            ← 重新加锁
if (oldTerm != this.currTerm) return  ← ABA 检查！
```

为什么要这样做？

因为 `getLastLogId()` 可能涉及磁盘 IO（如果日志不在内存中），我们不能持着 `writeLock` 去做 IO——那会阻塞所有其他操作。所以先 unlock，做完 IO 再 relock。

但在 unlock 的这段时间里，**可能有其他线程修改了 `currTerm`**——比如收到了一个更高 term 的 RPC 消息，触发了 `stepDown(newTerm)`。这时候我们基于旧 term 获取的 `lastLogId` 就不能用了——因为上下文已经变了。

`oldTerm != this.currTerm` 就是在检查"我 unlock 的这段时间里，有没有人改了 term"。如果改了，直接放弃当前操作——这就像你正在填一张表格，突然有人告诉你"规则改了"，你不能用旧规则填好的表格继续交——得撕掉重来。

### volatile 与 ReadWriteLock 的配合

前面讲 01-overview 的时候提到过：`state` 是 `volatile`，但 `currTerm` 不是。原因是：

- `state` 有时候在**不加锁**的情况下被快速检查（比如 `handleElectionTimeout()` 开头的快速路径 `if (this.state != STATE_FOLLOWER) return`）。`volatile` 保证这种无锁读能看到最新值
- `currTerm` 的所有读写都在 `readLock` 或 `writeLock` 保护下，锁本身就保证了可见性，不需要额外的 `volatile`

这种"该 volatile 的 volatile，不该 volatile 的不 volatile"的精细控制，是 JRaft 并发设计的一个特点——**不做多余的同步，但也绝不遗漏。**

---

## 总结：选举的"灵魂"

让我们用一张流程图来总结整个选举过程：

```
Follower 超时
    │
    ▼
PreVote（试探）
    │ "大家觉得需要选举吗？"
    │ 用 term+1 试探，不真正增加 term
    │
    ├── 多数派拒绝 → 放弃（Leader 还活着，不需要选举）
    │
    └── 多数派同意 ↓
        │
        ▼
    electSelf（正式竞选）
        │ currTerm++
        │ votedId = self
        │ 持久化 votedFor
        │ 发 RequestVote RPC
        │
        ├── 收到更高 term → stepDown（退出）
        │
        ├── 多数派拒绝 → voteTimer 超时后重试
        │
        └── 多数派同意 ↓
            │
            ▼
        becomeLeader（上任）
            │ 启动 Replicator
            │ 提交 no-op 日志
            │ 启动 stepDownTimer
```

选举的几个核心设计原则：

1. **PreVote 防止 term 污染**：网络隔离的节点不会无谓地增加 term
2. **日志新旧比较保证安全**：只有日志足够新的节点才能当选，保证已提交的数据不丢
3. **同 term 只投一票 + 持久化**：防止一个 term 里出现两个 Leader
4. **随机超时防止选票分裂**：确保选举最终能成功
5. **优先级衰减兼顾活性**：高优先级优先，但高优先级挂了也不至于选不出来
6. **ABA 防御保证并发安全**：unlock 期间 term 变化了就放弃，重来

你现在应该能理解为什么 Raft 论文把选举叫做"最重要的安全性保证"了——**选举选错了，一切都完了。**

好了，下一篇我们聊**日志复制**——Leader 选出来了，它的第一个任务就是把日志复制给所有 Follower。这也是 Raft 性能优化最集中的地方（Pipeline、批量发送、并行写入……），绝对精彩。
