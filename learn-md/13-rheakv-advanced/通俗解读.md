# 用人话聊聊 RheaKV：在 Raft 引擎上造一辆汽车

> 这篇文档是 [13-rheakv-advanced/README.md](./README.md) 的通俗伴读版。

---

## 开场：JRaft 是引擎，RheaKV 是整辆车

前面 12 篇，我们把 JRaft 这个"共识引擎"拆了个底朝天。但 JRaft 本身不是一个可以直接用的数据库——它只负责"让多台机器在一组操作上达成共识"。

**RheaKV 是在 JRaft 这个引擎上面，加了分片、路由、自动分裂、故障转移等"外壳"，造出来的一辆可以直接上路的汽车。**

---

## 第一幕：为什么需要分片？

单个 Raft Group 有一个天然限制：**所有数据都在同一组节点上**。如果你有 1TB 的数据，每个节点都得存 1TB——成本太高，而且单机 IO 可能成为瓶颈。

**分片（Sharding）** 的思路是：把数据按 key 的范围切成很多片（Region），每个片用一个独立的 Raft Group 来管理。

举个例子：

```
Region-1: [a, m)  → Raft Group 1 (节点 A, B, C)
Region-2: [m, z)  → Raft Group 2 (节点 D, E, F)
```

key=`foo` 落在 Region-1（因为 `f` 在 `[a, m)` 范围内），key=`zoo` 落在 Region-2。不同 Region 的数据由不同的 Raft Group 管理，**读写完全独立、互不干扰**。

这就像图书馆分楼层——A-M 的书在 1 楼，N-Z 的书在 2 楼。你找 `foo` 就去 1 楼，找 `zoo` 就去 2 楼。

---

## 第二幕：核心组件——"图书馆的分工"

### StoreEngine——"一整层楼"

`StoreEngine` 是一个物理节点上所有 Region 的管理器。一台机器上可能有多个 Region（就像一层楼上有多个书架），`StoreEngine` 管理所有这些 Region 的启动、停止、拆分。

### RegionEngine——"一个书架的管理员"

每个 Region 对应一个 `RegionEngine`。它包含：
- 一个 `RaftGroupService`（一个完整的 Raft Group）
- 一个 `MetricsRawKVStore`（本地 RocksDB 存储）

### RegionRouteTable——"索引卡片"

客户端怎么知道某个 key 在哪个 Region？通过 `RegionRouteTable`——一个路由表。

路由表的结构很简单：一个 `TreeMap<byte[], Region>`，key 是 Region 的 `startKey`。查找某个 key 属于哪个 Region：`floorEntry(key)` → O(log n)。

### PlacementDriver（PD）——"图书馆馆长"

PD 是 RheaKV 的"大脑"，负责全局调度：
- 维护集群拓扑：哪些 Store 上有哪些 Region
- 自动分裂：Region 数据量太大 → 一分为二
- 负载均衡：某个 Store 负载太高 → 迁移部分 Region 到空闲 Store
- Leader 平衡：某个 Store 上 Leader 太多 → Transfer Leadership

## 第三幕：一次写请求的完整旅程

```
客户端：put("foo", "bar")
    │
    ▼
① RegionRouteTable：foo 在哪个 Region？
    → Region-1 [a, m)，Leader 是节点 A
    │
    ▼
② 发送请求到节点 A 的 Region-1
    │
    ▼
③ 节点 A 的 RaftGroupService：
    → NodeImpl.apply(put("foo","bar"))
    → 日志复制到 B、C
    → 多数派确认
    │
    ▼
④ 状态机执行：
    → RocksDB.put("foo", "bar")
    → 回调客户端："成功了"
```

如果 Region-1 的 Leader 不在节点 A 上（路由表过时了），客户端会收到一个"Leader Changed"错误，然后刷新路由表，重新发送请求。

---

## 第四幕：自动分裂——"书架太满了怎么办"

Region 的数据量是会增长的。当一个 Region 的数据量超过阈值（默认 512MB），PD 会触发**自动分裂（Split）**。

分裂的过程：

1. PD 决定在 `splitKey` 处分裂（通常是 Region 中间的某个 key）
2. Leader 写一条"分裂日志"（配置变更日志）
3. 所有节点执行分裂：Region-1 `[a, m)` → Region-1 `[a, g)` + Region-3 `[g, m)`
4. 更新路由表

分裂后，两个新 Region 各管一半的 key 范围。如果继续增长，还会继续分裂——这就是"自动分片"的核心。

### 分裂为什么安全？

因为分裂操作本身也是通过 Raft 日志来执行的——Leader 写分裂日志、复制到多数派、所有节点一起执行分裂。这保证了**所有副本的分裂结果一致**。

---

## 第五幕：批量操作与多 Region 散射

客户端可能一次性写入多个 key：`multiPut({foo: bar, zoo: baz})`。

但 `foo` 和 `zoo` 可能在不同的 Region 里！RheaKV 的处理是**散射-聚合（Scatter-Gather）**：

1. 把 keys 按 Region 分组：`foo → Region-1`，`zoo → Region-2`
2. 并行发送请求到两个 Region
3. 等待两个 Region 都返回成功
4. 聚合结果，返回给客户端

**注意：跨 Region 的操作没有事务保证！** 可能 Region-1 成功、Region-2 失败。如果需要跨 Region 事务，需要额外的分布式事务机制（RheaKV 本身不提供）。

---

## 第六幕：故障转移

### Follower 挂了

不影响。只要多数派还在，Raft Group 继续工作。挂掉的 Follower 恢复后，通过日志复制追赶。

### Leader 挂了

Raft 自动选出新 Leader。客户端发现请求失败后，刷新路由表，找到新 Leader 重试。

### 整个 Store 挂了

这台机器上的所有 Region 都不可用。如果每个 Region 在其他 Store 上还有副本（Raft 多副本保证），这些 Region 会在其他 Store 上选出新 Leader，服务继续。

---

## 总结

RheaKV 的核心设计：

| 设计 | 解决的问题 |
|------|-----------|
| Region 分片 | 突破单机存储和性能限制 |
| RegionRouteTable | 快速路由 key → Region |
| PlacementDriver | 全局调度（分裂、均衡、迁移） |
| 自动分裂 | 数据量增长时自动分片 |
| Scatter-Gather | 跨 Region 批量操作 |
| Raft 多副本 | 容错、故障自动转移 |

**一句话总结**：RheaKV = JRaft（共识引擎）+ 分片（Region）+ 路由（RouteTable）+ 调度（PD）。它把 Raft 从"一个 Group 的共识"升级为"整个集群的分布式存储"。
