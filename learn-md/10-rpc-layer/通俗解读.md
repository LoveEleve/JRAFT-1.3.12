# 用人话聊聊 RPC 通信层：集群的"电话系统"

> 这篇文档是 [10-rpc-layer/README.md](./README.md) 的通俗伴读版。

---

## 开场：没有 RPC，Raft 就是自言自语

前面讲的选举、日志复制、快照安装——这些操作有一个共同前提：**节点之间能互相"说话"**。Leader 让 Follower 复制日志，Candidate 请大家投票，Leader 给落后者发快照——这些全靠 RPC。

**如果 Raft 协议是剧本，RPC 就是演员之间的对讲机。** 没有对讲机，再好的剧本也演不起来。

---

## 第一幕：RPC 层的核心职责

RPC 层需要做三件事：

### 1. 收消息（RpcServer）

每个节点启动一个 RpcServer，监听端口。其他节点发来的请求（投票、日志复制、快照安装……）都通过这个端口进来。

RpcServer 收到请求后，根据请求类型分发给不同的 `Processor`——就像前台接线员，把电话转接给对应的部门。

### 2. 发消息（RaftClientService）

每个节点也能主动给其他节点发消息。比如 Leader 发 `AppendEntries`、Candidate 发 `RequestVote`。

RaftClientService 封装了所有"发消息"的操作，底层使用连接池复用 TCP 连接。

### 3. Pipeline 有序响应

日志复制走的是 Pipeline 模式——Leader 连续发多个 AppendEntries 请求，不等前一个的响应就发下一个。但 Follower 的响应必须按**请求顺序**返回。

为什么要有序？因为 Replicator 依赖响应顺序来推进 `nextIndex`。如果响应乱序了，`nextIndex` 的推进就会出错。

JRaft 在 RPC 层面实现了 Pipeline 有序响应：每个 `<Leader, Follower>` 对维护一个请求队列，Follower 必须按入队顺序发送响应。

## 第二幕：可插拔——SPI 机制

JRaft 支持两种 RPC 实现：

1. **SOFABolt**（默认）：蚂蚁自研的高性能 RPC 框架，基于 Netty
2. **gRPC**：Google 的通用 RPC 框架

怎么切换？通过 **SPI（Service Provider Interface）**。

SPI 的核心思想是"面向接口编程"。JRaft 定义了两个接口：
- `RaftRpcFactory`：创建 RpcClient 和 RpcServer
- `RpcRequestClosure`：处理 RPC 请求和响应

不同的实现（Bolt 和 gRPC）各自提供这些接口的实现类。JRaft 在启动时通过 `ServiceLoader` 加载配置好的实现——改一个 `META-INF/services` 文件就能切换 RPC 框架，**业务代码一行不改**。

这就像手机换 SIM 卡——换了运营商（Bolt→gRPC），但拨打电话的方式（API）不变。

### Bolt vs gRPC

| 特性 | SOFABolt | gRPC |
|------|----------|------|
| 性能 | 极高（为蚂蚁内部优化） | 高（通用优化） |
| 协议 | 自定义二进制协议 | HTTP/2 + Protobuf |
| 跨语言 | ❌（Java only） | ✅（多语言支持） |
| 生态 | 蚂蚁内部 | 全球通用 |
| 默认 | ✅ | ❌ |

如果你在纯 Java 环境且追求极致性能 → Bolt。如果需要跨语言或与云原生生态集成 → gRPC。

## 第三幕：线程隔离——急诊通道和普通通道分开

Raft 的 RPC 分两类：

1. **核心 RPC**：`AppendEntries`、`RequestVote`、`InstallSnapshot`——这些是 Raft 协议的"心跳"，必须低延迟
2. **管理 RPC**：`AddPeer`、`RemovePeer`、`TransferLeader`、`GetLeader`——这些是运维操作，频率低但可能耗时

JRaft 用**不同的线程池**处理这两类 RPC：

- 核心 RPC 用 `RAFT_RPC_EXECUTOR`（默认核心线程数 = CPU 核数 × 6）
- 管理 RPC 用 `CLI_RPC_EXECUTOR`（默认核心线程数 = CPU 核数）

这样即使运维操作（如大规模成员变更）把管理线程池打满了，核心的日志复制和心跳也不受影响。

**这就像医院的急诊通道和普通挂号通道分开——普通门诊排再长的队，急诊都能第一时间处理。**

---

## 第四幕：AppendEntries 的特殊待遇

在所有 RPC 中，`AppendEntries` 是最特殊的——它是 Raft 的"热路径"，每条日志都要走一次。JRaft 对它做了很多特殊优化：

### 1. 请求/响应缓存

`AppendEntriesRequest` 和 `AppendEntriesResponse` 是最频繁的消息。JRaft 使用 Protobuf 序列化，并且复用消息对象（Builder 模式），减少 GC 压力。

### 2. Pipeline 有序响应

Follower 收到 AppendEntries 后，必须按**接收顺序**返回响应。JRaft 在 Processor 内部维护了一个 `PeerRequestContext`，每个请求分配一个序号，响应按序号排队发送。

### 3. 心跳优先

Leader 的心跳也是 AppendEntries（不带日志的空请求）。心跳超时是 Raft 的生命线——超时了 Follower 就会发起选举。

JRaft 的处理器 `AppendEntriesRequestProcessor` 对心跳请求做了快速路径——检测到 `entriesCount == 0`（心跳）时，跳过一些不必要的处理逻辑，直接响应。

---

## 第五幕：连接管理与故障检测

### 连接池

JRaft 使用连接池复用 TCP 连接。多个 RPC 请求共享同一个 TCP 连接，避免频繁建立/断开连接的开销。

### 故障检测

RPC 层会检测连接异常：
- **连接断开**：Replicator 收到 `ECONNRESET` 错误，触发重连
- **响应超时**：`invokeAsync` 设置了超时时间，超时后回调 `onError`
- **对端不可达**：DNS 解析失败或端口未监听，直接失败

这些错误最终会传导到 Replicator，Replicator 会决定是否重试、是否切换到 Snapshot 模式。

---

## 总结

RPC 层的核心设计：

| 设计 | 解决的问题 |
|------|-----------|
| RpcServer + Processor 分发 | 不同类型的请求由不同处理器处理 |
| RaftClientService | 封装所有"发消息"操作 |
| Pipeline 有序响应 | 日志复制请求的响应保持顺序 |
| SPI 可插拔 | Bolt/gRPC 自由切换 |
| 线程池隔离 | 管理操作不影响核心 Raft RPC |
| 连接池 | 复用 TCP 连接，减少开销 |
| 心跳快速路径 | 心跳延迟最小化 |

**一句话总结**：RPC 层是 Raft 的"神经系统"——负责节点间所有通信，性能和可靠性直接决定了集群的健康状况。
