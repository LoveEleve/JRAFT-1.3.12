## 开场：日志不存磁盘，就等于没记

前面我们花了大量篇幅聊"日志复制"——Leader 怎么把日志发给 Follower。但这些日志最终要落到磁盘上。为什么？因为**内存是易失性存储**——服务器一断电，内存里的数据全没了。

Raft 的核心承诺是"只要数据被多数派确认了，就不会丢"。这个承诺的前提是——**每个节点的日志都持久化到了磁盘上**。如果日志只在内存里，节点一宕机就没了，"多数派确认"就是一句空话。

所以日志存储层虽然不像选举和复制那么"戏剧性"，但它是**整个 Raft 正确性的基石**。

---

## 第一幕：为什么选 RocksDB？

JRaft 默认用 **RocksDB** 来存日志。RocksDB 是 Facebook 开源的嵌入式 KV 数据库，基于 Google 的 LevelDB 改进而来。

**为什么不用 MySQL、Redis，甚至直接写文件？**

让我们从日志存储的需求出发：

### 需求 1：写入要快

Raft 的日志是**追加写**（append-only）——新日志只会往后面加，不会修改已有的日志。这就非常适合 RocksDB 的 **LSM-Tree**（Log-Structured Merge-Tree）架构。LSM-Tree 把所有写操作先写到内存的 MemTable 里（超快），然后再异步合并（Compaction）到磁盘上。写入性能远超 B-Tree 架构的数据库（如 MySQL InnoDB）。

### 需求 2：按 index 查询要快

Replicator 复制日志时，需要频繁按 `index` 读取某条日志。RocksDB 支持 `get(key)` 操作——O(1) 级别（在 MemTable 或 Block Cache 命中时）。

JRaft 的设计是：**key = index 的 8 字节大端序编码，value = 序列化后的 LogEntry**。因为 RocksDB 的 key 是按字节序排序的，大端序编码保证了数字大小与字节序一致——index=1 的 key 在 index=2 的前面。

### 需求 3：支持批量删除

快照安装后，旧日志要删掉（`truncatePrefix`）。Leader 切换时，未提交的日志要回滚（`truncateSuffix`）。RocksDB 的 `deleteRange` 可以高效地删除一个范围内的所有 key。

### 需求 4：嵌入式，不需要额外部署

RocksDB 是嵌入式数据库——作为一个 Java 库直接嵌入到 JRaft 进程里，不需要额外部署和维护。这比 MySQL（需要独立部署）方便得多。

### 总结一下

| 需求 | RocksDB | MySQL | 直接写文件 |
|------|---------|-------|-----------|
| 追加写性能 | ⭐⭐⭐⭐⭐ (LSM-Tree) | ⭐⭐⭐ (B-Tree) | ⭐⭐⭐⭐ |
| 按 index 查询 | ⭐⭐⭐⭐ (get) | ⭐⭐⭐⭐ (主键查询) | ⭐⭐ (需自建索引) |
| 范围删除 | ⭐⭐⭐⭐ (deleteRange) | ⭐⭐⭐ (DELETE) | ⭐ (很难) |
| 嵌入式 | ✅ | ❌ | ✅ |
| 原子批量写 | ✅ (WriteBatch) | ✅ (事务) | ❌ (需自己实现) |

RocksDB 几乎完美符合 Raft 日志存储的所有需求。

## 第二幕：两张表——default CF 和 conf CF

RocksDB 有一个概念叫 **Column Family（CF）**——你可以把它理解为"一个 RocksDB 数据库里的两张表"。

JRaft 创建了两个 CF：

### default CF（默认表）

存储**所有日志**（包括普通日志和配置日志）。

```
Key:   8字节大端序 long（logIndex）
Value: LogEntryEncoder.encode(entry)
```

### conf CF（配置表）

**只存储配置日志**和一个特殊的元数据 key。

配置日志是什么？就是记录"集群成员变更"的日志——比如"节点 D 加入集群"或"节点 A 退出集群"。

**为什么配置日志要单独存一份？**

因为节点启动时，需要快速恢复集群的配置信息——"现在集群里有谁？"如果配置日志和普通日志混在一起，就得扫描整个日志文件才能找到所有配置变更记录。日志量大的时候（几百万条），这个扫描会非常慢。

把配置日志单独存到 conf CF 里，启动时只扫描 conf CF 就行了——配置日志通常只有几十条，瞬间完成。

**注意：配置日志在两个 CF 里都有副本。** default CF 里的副本保证日志 index 的连续性（Replicator 按 index 复制时不会跳过配置日志），conf CF 里的副本方便快速恢复。

## 第三幕：写入——WriteBatch 的原子性保证

当 Leader 要写入一批日志时，JRaft 使用 RocksDB 的 **WriteBatch**：

```java
WriteBatch batch = new WriteBatch();
for (LogEntry entry : entries) {
    if (entry.type == CONFIGURATION) {
        batch.put(defaultHandle, key, value);  // 写 default CF
        batch.put(confHandle, key, value);     // 也写 conf CF
    } else {
        batch.put(defaultHandle, key, value);  // 只写 default CF
    }
}
db.write(writeOptions, batch);  // 原子提交！
```

**WriteBatch 保证原子性——要么全部写入成功，要么全部失败。** 不会出现"写了一半"的情况。这对 Raft 非常重要——如果一批日志写了一半就失败了，后续的日志复制和提交逻辑就会出错。

### 读写锁的"反用"

一个很有意思的设计：`appendEntries()` 用的是**读锁**（`readLock`），而不是写锁。

"写日志怎么用读锁？这不是乱来吗？"

别急。这里的读写锁不是用来保护"日志数据"的——RocksDB 的 WriteBatch 本身就是线程安全的。这里的读写锁是用来保护"RocksDB 实例的生命周期"的：

- `readLock`：表示"我在使用 RocksDB"——多个线程可以同时使用（并发写入）
- `writeLock`：表示"我要关闭 RocksDB"——独占，确保没有人在使用时才能关闭

所以 `appendEntries()` 持 `readLock`，`shutdown()` 持 `writeLock`。这样多个 `appendEntries()` 可以并发执行（提升吞吐量），而 `shutdown()` 会等所有正在执行的 `appendEntries()` 完成后才关闭 RocksDB。

## 第四幕：删除——truncatePrefix 和 truncateSuffix

### truncatePrefix（删除旧日志）

快照安装后，快照覆盖的旧日志可以删掉了。

JRaft 的做法是**先持久化新的 firstLogIndex，再异步删除**：

1. 把新的 `firstLogIndex` 写到 conf CF 的一个特殊 key 里
2. 更新内存中的 `firstLogIndex`
3. 提交一个后台任务，异步删除旧日志

**为什么先持久化 firstLogIndex？**

假设顺序反过来：先删除日志，再持久化 firstLogIndex。如果删除完了、还没来得及持久化 firstLogIndex 就崩溃了——重启后，`firstLogIndex` 还是旧值，但旧日志已经被删了。这时候尝试读取旧日志就会发现"日志不存在"——**数据不一致！**

先持久化 firstLogIndex，即使删除过程中崩溃了也没关系——重启后 firstLogIndex 已经是新值了，不会去读已删除的旧日志。

### truncateSuffix（回滚未提交日志）

Leader 切换后，新 Leader 可能发现 Follower 有一些"前任 Leader 写的、但没有被提交的"日志——这些日志需要被覆盖。

`truncateSuffix` 是**同步删除**的（不是异步）——因为必须在 Leader 发送新日志之前把旧日志删掉，否则会出现日志冲突。

## 第五幕：编码——V1 和 V2 格式

日志写入 RocksDB 之前，需要序列化成 `byte[]`。JRaft 支持两种编码格式：

### V1 格式（已废弃）

直接用 Protobuf 序列化。简单粗暴，但没有格式标识——你拿到一个 `byte[]`，不知道它是 V1 还是其他格式。

### V2 格式（当前默认）

在 Protobuf 前面加了一个 6 字节的 Header：

```
[0xBB][0xD2][0x01][0x00][0x00][0x00][Protobuf数据...]
 ←Magic→    ←Ver→ ←──Reserved──→
```

**Magic Bytes（`0xBB 0xD2`）** 就像文件的"签名"——看到这两个字节就知道"这是 V2 格式"。

**为什么需要 Magic Bytes？** 因为滚动升级时，集群里可能同时有 V1 和 V2 格式的日志。`AutoDetectDecoder` 检查第一个字节：如果是 `0xBB` 就用 V2 解码，否则用 V1。**一个字节就能区分两种格式——简单高效。**

## 第六幕：元数据存储——LocalRaftMetaStorage

除了日志，还有两个关键信息需要持久化：

1. **currTerm**：当前任期
2. **votedFor**：当前任期投给了谁

这两个信息不存在 RocksDB 里——它们有自己专属的存储：`LocalRaftMetaStorage`。

### 为什么不放 RocksDB？

两个原因：

1. **解耦**：`reset()` 会销毁整个 RocksDB。如果 `currTerm` 和 `votedFor` 也在 RocksDB 里，`reset()` 就会把它们也删掉——这是灾难性的（节点忘记了 currTerm 和 votedFor）
2. **简单**：就两个字段，用 Protobuf 文件存就够了，不需要 KV 数据库的复杂性

### 实现

`LocalRaftMetaStorage` 在磁盘上是一个叫 `raft_meta` 的 Protobuf 文件。每次修改 `currTerm` 或 `votedFor`，都会调用 `save()` 同步写入文件。

"同步写入"意味着这是一个**阻塞操作**——调用方（NodeImpl）必须等写完才能继续。这会影响性能吗？会。但这是**安全性要求**——Raft 协议要求 `votedFor` 必须在投票**之前**持久化，不能先投票再持久化（万一投完没持久化就崩溃，重启后又投一次，可能导致双 Leader）。

## 第七幕：大 Value 优化——RocksDBSegmentLogStorage

RocksDB 虽好，但它有一个弱点：**对大 Value 不友好**。

RocksDB 的 LSM-Tree 会定期做 Compaction（合并文件）。Compaction 时需要读出旧文件、合并、写入新文件。如果 Value 很大（比如几 MB），Compaction 时的读写放大会非常严重——CPU 和 IO 都会飙高。

JRaft 提供了一个解决方案：**RocksDBSegmentLogStorage**。

思路是"索引和数据分离"——就像图书馆的索引卡和书架：

- **索引卡（RocksDB）**：存的是 `(segmentFileOffset, length)`——"这条日志在第几个 Segment 文件的第几个字节开始，长度多少"
- **书架（Segment 文件）**：存的是实际的日志数据，顺序写入

Compaction 时，RocksDB 只需要合并 `(offset, length)` 这种小 Value，不需要搬动实际的日志数据。

### 模板方法模式

`RocksDBSegmentLogStorage` 继承 `RocksDBLogStorage`，通过覆盖 6 个模板方法来实现"数据外置"：

- `onDataAppend()`：写数据时，不写入 RocksDB，而是写入 Segment 文件，返回 `(offset, length)`
- `onDataGet()`：读数据时，从 RocksDB 读出 `(offset, length)`，再从 Segment 文件读取实际数据
- `onTruncatePrefix()`、`onTruncateSuffix()`：删除时，同时清理 Segment 文件
- `onSync()`：刷盘时，除了 RocksDB 的 sync，还要 sync Segment 文件

这是一个教科书级的**模板方法模式（Template Method Pattern）** 应用——基类定义骨架，子类覆盖具体步骤。你不需要改基类的一行代码，就能切换存储策略。

## 总结

日志存储层看似平淡，但暗藏很多精妙的设计：

| 设计 | 解决的问题 |
|------|-----------|
| RocksDB | 追加写性能好、支持按 index 查询、支持范围删除 |
| 两个 Column Family | 配置日志快速恢复，不需要扫描全量日志 |
| WriteBatch | 原子写入，不会出现"写了一半" |
| truncatePrefix 先持久化 firstLogIndex | 崩溃安全，不会读到已删除的日志 |
| V2 编码 + AutoDetect | 兼容新旧格式，支持滚动升级 |
| LocalRaftMetaStorage | 元数据与日志解耦，reset() 不会影响 term/votedFor |
| RocksDBSegmentLogStorage | 大 Value 场景下避免 Compaction 放大 |
| 读写锁"反用" | 允许并发写入，同时保护 shutdown |

**一句话总结**：日志存储层的核心目标就是——**把日志安全、高效、可靠地存到磁盘上，并且能快速读出来。**
