# 用人话聊聊并发基础设施：JRaft 性能的秘密武器

## 开场：为什么要专门讲"工具"？

前面 10 篇讲的是 JRaft "做什么"（选举、复制、快照……），这一篇讲的是 JRaft "用什么做"——底层的并发工具箱。

你可能觉得"工具有什么好讲的"——用 `synchronized` 加个锁不就完了？

**在高性能系统里，工具的选择直接决定了系统的上限。** 就像赛车和家用车的区别不在于谁的方向盘更圆，而在于发动机、变速箱、悬挂这些"底层零件"的性能差异。

JRaft 能在蚂蚁的超大规模场景下跑出高性能，很大程度上归功于这些精心选择的并发工具。

---

## 第一幕：Disruptor——"寿司旋转传送带"

### 为什么不用 BlockingQueue？

Java 自带的 `LinkedBlockingQueue` 底层用 `ReentrantLock`——生产者和消费者各一把锁。在高并发下，多个生产者线程竞争同一把锁，会导致大量的上下文切换和缓存行失效。

Disruptor 是一个**无锁的环形缓冲区**。它的核心设计是：

1. **环形数组（RingBuffer）**：预分配固定大小的数组，对象复用，几乎零 GC
2. **CAS 序号竞争**：生产者通过 CAS 竞争序号，不需要锁
3. **缓存行填充**：关键字段前后用无意义的 long 填充，防止"伪共享"（两个不相关的变量在同一个 CPU 缓存行中，一个变量修改导致另一个变量的缓存失效）

### JRaft 里哪些地方用了 Disruptor？

至少 4 个地方：

| 组件 | 用途 | 生产者 | 消费者 |
|------|------|--------|--------|
| LogManager | 异步写磁盘 | NodeImpl.apply() | 磁盘写线程 |
| FSMCaller | 异步 apply 日志 | BallotBox.commitAt() | FSM 线程 |
| ReadOnlyService | 异步处理读请求 | Node.readIndex() | ReadIndex 线程 |
| NodeImpl | 异步 apply 任务 | 客户端线程 | apply 线程 |

### 为什么 Disruptor 的大小必须是 2 的幂？

因为 `index % size` 运算在 size 是 2 的幂时可以用**位运算** `index & (size - 1)` 代替——位运算比取模运算快得多。这是一个微小但在高频调用下非常有效的优化。

## 第二幕：RepeatedTimer——"智能闹钟"

Raft 协议依赖大量的定时器：
- `electionTimer`：选举超时定时器
- `heartbeatTimer`：心跳定时器（Leader 用）
- `snapshotTimer`：快照定时器
- `stepDownTimer`：Leader 定期检查"我还能联系到多数派吗"
- `voteTimer`：投票超时定时器

这些定时器有几个共同需求：
1. **可重置**：收到心跳后要重置选举定时器
2. **可暂停/恢复**：竞选期间暂停选举定时器
3. **随机抖动**：选举超时需要随机化（防止选票分裂）
4. **线程安全**：多个线程可能同时操作定时器

`RepeatedTimer` 封装了 `HashedWheelTimer`（Netty 的时间轮定时器），在上面加了状态机（Started/Stopped/Destroyed）和随机抖动支持。

### HashedWheelTimer vs ScheduledExecutorService

| 特性 | HashedWheelTimer | ScheduledExecutorService |
|------|-----------------|--------------------------|
| 定时精度 | 毫秒级（依赖 tick） | 纳秒级 |
| 大量定时任务 | 高效（O(1) 添加） | 低效（优先级队列 O(log n)） |
| 适用场景 | 大量粗精度定时（如超时检测） | 少量精确定时 |

Raft 的定时器不需要纳秒精度——选举超时动辄几百毫秒，误差几毫秒无所谓。但定时器数量可能很多（每个 Raft Group 5 个定时器），用 `HashedWheelTimer` 更高效。

## 第三幕：SegmentList——"分段书架"

### ArrayList 的痛点

`ArrayList` 在两种操作上性能很差：

1. **扩容**：超出容量时，分配新数组 + 拷贝所有元素 → O(n)，可能触发 STW（Stop-The-World）般的延迟尖刺
2. **头部删除**：`remove(0)` 需要把所有元素往前移一位 → O(n)

而 JRaft 的日志内存缓存恰好需要这两种操作：
- 新日志不断追加 → 需要频繁扩容
- 旧日志需要清除（`truncatePrefix`） → 需要头部删除

### SegmentList 的设计

`SegmentList` 把数据分成一段一段的小数组（每段 128 或 256 个元素）：

```
segments: [seg0: [e0..e127]] [seg1: [e128..e255]] [seg2: [e256..e383]]
           ↑ firstOffset=0
```

- **追加**：往最后一个 segment 的末尾追加。segment 满了？再加一个新 segment。不需要拷贝旧数据
- **头部删除**：删前 128 个元素？直接把 seg0 整个丢弃。O(1)！
- **随机访问**：`get(index)` → `segments[index / segmentSize][index % segmentSize]`。两次数组访问，O(1)

### 性能对比

| 操作 | ArrayList | SegmentList |
|------|-----------|-------------|
| 追加（无扩容） | O(1) | O(1) |
| 追加（扩容） | O(n) 拷贝 | O(1) 新建 segment |
| 头部批量删除 | O(n) 移动 | O(n/segmentSize) 丢弃 segment |
| 随机访问 | O(1) | O(1)（两次数组访问） |

SegmentList 在"追加 + 头部删除"的场景下完胜 ArrayList。

## 第四幕：NonReentrantLock——"防误用锁"

Java 的 `ReentrantLock` 允许同一个线程多次加锁（重入）。这在大多数场景下是好的——但在 JRaft 的某些场景下，重入说明代码有 bug。

比如 Replicator 的操作不应该递归调用——如果你在持锁期间又调了一个会加锁的方法，说明调用链有问题（可能会导致逻辑混乱或性能问题）。

`NonReentrantLock` 的实现很简单：基于 `AbstractQueuedSynchronizer`（AQS），state=0 表示未锁，state=1 表示已锁。`tryAcquire` 时如果 state=1 **直接返回 false**（而不是检查持有者线程）。

**这是一个"开发阶段的安全护栏"**——帮你在第一时间发现逻辑错误，而不是等到生产环境才暴露。

## 第五幕：其他重要工具

### StampedLock（BallotBox 用）

比 `ReentrantReadWriteLock` 更快的读写锁。支持**乐观读**——先不加锁就读，读完后验证期间是否有写操作。大部分情况下验证通过（没有写），省去了加锁开销。

在 BallotBox 中，`getLastCommittedIndex()` 被非常频繁地调用（每次心跳、每次 AppendEntries），乐观读让这个高频操作几乎零锁开销。

### CountDownEvent（SnapshotExecutor 用）

类似 `CountDownLatch`，但可以**动态增减计数**。用于等待"所有快照任务完成"——`incrementAndGet()` 标记一个新任务开始，`countDown()` 标记一个任务完成，`await()` 等待所有任务完成。

### ThreadId（Replicator 的唯一标识）

Replicator 不是直接用对象引用来标识的——而是用一个 `ThreadId`。`ThreadId` 像一个"智能指针"：
- `lock()` 获取 Replicator 引用并加锁
- `unlock()` 释放锁
- 如果 Replicator 已被销毁，`lock()` 返回 null

这避免了"持有一个已被销毁的 Replicator 引用"的问题。

## 总结

JRaft 的并发工具箱：

| 工具 | 用途 | 核心优势 |
|------|------|---------|
| Disruptor | 异步事件队列 | 无锁、零 GC、极高吞吐 |
| HashedWheelTimer | Raft 定时器 | 大量定时任务 O(1) 管理 |
| SegmentList | 日志内存缓存 | 追加 + 头删均为 O(1) |
| NonReentrantLock | 防误用 | 开发阶段发现递归加锁 bug |
| StampedLock | BallotBox 读写 | 乐观读零锁开销 |
| CountDownEvent | 等待异步任务 | 动态增减计数 |
| ThreadId | Replicator 标识 | 安全的"智能指针" |

**一句话总结**：高性能不是来自"算法有多精妙"，而是来自"每一个细节都用了最合适的工具"。JRaft 在并发工具的选择上做到了极致。
